# JetCheck 1.4 一期测量项分层与命名建议 v0.1

本文档用于基于基恩士 TM-X Navigator 与当前团队已有测量工具截图，整理 1.4 一期更适合采用的测量项分层与命名方式。

这份材料的目标不是一次性定死所有测量类型，而是先回答下面 3 个问题：

1. 我们的一期工具栏顶层分层应该怎么收
2. 测量项名称应该按什么口径命名
3. TM-X 里的大量工具，哪些值得一期参考，哪些不适合直接照搬

## 1. 当前观察

基于这轮截图，我的判断是：

1. TM-X 的能力很全，但工具颗粒度非常细，已经带有明显专家工具属性。
2. 你们现有工具的顶层分层更轻，用户心智也更直观。
3. 一期真正需要的不是“工具数量很多”，而是“分层清楚、命名稳定、后续能扩”。

可以把两边的差异简单理解成：

| 参考对象 | 特征 | 对 1.4 的启发 |
| --- | --- | --- |
| TM-X | 工具很多、分组很细、应用型能力强 | 适合做能力参考，不适合直接照抄信息架构 |
| 现有工具 | 分层更轻、路径更短、易理解 | 更适合作为 1.4 一期产品骨架 |

## 2. 一期建议保留的顶层分层

我建议一期继续沿用你们现有工具的 5 个顶层分层，而不是改成 TM-X 那种多层复杂工具箱：

1. `元素`
2. `构造`
3. `测量`
4. `公差`
5. `检测`

原因：

1. 这 5 层符合“先提元素，再做关系，再出结果，再判定”的自然路径。
2. 用户更容易理解自己当前是在“找元素”“做辅助构造”还是“定义输出结果”。
3. 后续即使扩展更多能力，也可以在这 5 层内继续长，不需要推翻整体结构。

## 3. 每一层到底负责什么

### 3.1 元素

用于从图像中提取后续测量要用到的基础几何对象。

建议理解为：

1. 线
2. 圆
3. 点
4. 轮廓 / 边缘

产品原则：

1. 元素层只回答“图像里有哪些可用几何对象”。
2. 不在元素层直接输出业务测量结果。

### 3.2 构造

用于在已有元素基础上生成辅助几何关系。

建议理解为：

1. 交点
2. 中点
3. 中线 / 基准线
4. 平行线
5. 垂线

产品原则：

1. 构造层只负责生成辅助对象。
2. 不把“距离”“角度”这类最终结果型工具塞进构造层。

### 3.3 测量

用于定义最终要输出的数值结果。

建议理解为：

1. 距离
2. 直径 / 半径
3. 角度
4. 长度 / 宽度
5. 间隙

产品原则：

1. 测量层直接面向“结果”。
2. 命名尽量按用户看的对象关系来写，而不是按算法名字来写。

### 3.4 公差

用于定义测量结果的上下限和合格判定。

一期建议理解为：

1. 标称值
2. 上限
3. 下限
4. OK / NG

产品原则：

1. 一期先把测量项级别的公差判定做好。
2. 不急着一上来做完整几何公差体系。

### 3.5 检测

用于承接更高阶、面向具体业务场景的组合型能力。

一期建议：

1. 先轻放。
2. 更偏向结果展示和少量应用型能力入口。
3. 不把复杂业务综合判定塞进这里。

## 4. 命名建议

一期建议统一按“用户对象关系”命名，而不是按算法名命名。

推荐命名规则：

1. 距离类：`点点距离`、`点线距离`、`线线距离`、`圆圆距离`
2. 直径类：`圆直径`
3. 角度类：`线夹角`
4. 宽高类：`长度`、`宽度`
5. 间隙类：`间隙`

不建议的一期命名方式：

1. 过度算法化
   - 例如“二点欧氏距离”“轮廓最短距”
2. 过度底层化
   - 例如“元素 A 到元素 B 的法向投影距离”
3. 同类工具多套命名并存
   - 例如同时出现“点点距离”“两点距离”“点-点距离”

建议固定成一套统一格式：

1. 对象关系类：`对象1 + 对象2 + 结果`
2. 单对象测量类：`对象 + 结果`

## 5. TM-X 能力如何映射到我们的一期分层

TM-X 的很多能力可以参考，但建议按我们自己的层次重新归类。

| TM-X 常见分组 | 更适合映射到我们哪里 | 一期建议 |
| --- | --- | --- |
| 直线、圆、圆弧、轮廓、点 | `元素` | 可参考 |
| 中线、垂线、平行线、交点、中点 | `构造` | 可参考，但先收窄 |
| 距离、角度、直径、间隙 | `测量` | 一期优先参考 |
| 真直度、真圆度、同心度、平行度 | `公差` 或后续高级能力 | 一期不建议直接纳入 |
| 螺纹、批量操作、自动测量、缺陷距离 | `检测` 或后续应用工具 | 一期不建议直接纳入 |

## 6. 一期建议优先收进来的候选项

这里给的是“优先候选”，不是最终拍板版白名单。

### 6.1 元素层候选

1. 直线
2. 圆
3. 点
4. 轮廓

### 6.2 构造层候选

1. 交点
2. 中点
3. 中线 / 基准线

### 6.3 测量层候选

1. 点点距离
2. 点线距离
3. 线线距离
4. 圆直径
5. 圆圆距离

### 6.4 公差层候选

1. 标称值
2. 上限
3. 下限
4. OK / NG

### 6.5 检测层候选

1. 测量结果列表
2. 单图状态摘要
3. 异常项高亮

## 7. 一期暂不建议直接纳入的能力

基于当前阶段，我不建议一期直接纳入：

1. 完整几何公差体系
   - 真直度、真圆度、轮廓度、同心度、平行度
2. 过重的应用型封装工具
   - 螺纹、批量操作、自动测量
3. 缺陷检测和尺寸测量强耦合工具
   - 缺陷距离、异物测量一类复合能力
4. 太多近似 / 辅助 / 专家工具并列暴露
5. 角度类结果直接进入一期白名单

## 8. 和找正、ROI 的关系

这轮讨论里有一个很关键的边界，建议在文档里持续保持：

1. `搜索 ROI`
   - 定义去图像哪一块区域里找
2. `找正`
   - 在搜索 ROI 内基于参考特征做匹配，并支持一定范围的平移 / 旋转恢复
3. `元素`
   - 在找正后的图像坐标里提取线、圆、点、轮廓等对象
4. `测量`
   - 基于这些对象输出距离、直径、角度等结果

也就是说：

1. ROI 和找正不是“测量项类型”。
2. 它们是测量项运行前的基础定位能力。

## 9. 当前建议

如果按一期要“做得稳、做得成”，我建议先坚持下面这 5 条：

1. 顶层结构继续沿用 `元素 / 构造 / 测量 / 公差 / 检测`。
2. 工具命名优先按用户对象关系命名，不按算法名命名。
3. TM-X 作为能力参考，不直接照抄它的工具数量和分组方式。
4. 几何公差和应用型重工具先往后放。
5. 先把高频距离类和直径类能力收进一期候选，角度类先放到后续优先候选。
