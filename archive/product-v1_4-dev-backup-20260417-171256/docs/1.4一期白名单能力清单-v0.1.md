# JetCheck 1.4 一期白名单能力清单 v0.1

本文档用于把 1.4 一期能力收敛成一份可评审、可执行的白名单。

目标不是列“以后可能想做什么”，而是明确：

1. 1.4 一期确定要做什么
2. 1.4 一期建议先不做什么
3. 哪些能力需要继续确认，但必须在一期范围评审前收口

## 1. 白名单使用原则

1. 一期优先保证“最小完整闭环”，而不是功能面铺太宽。
2. 只有进入白名单的能力，才默认进入 1.4 正式方案与 demo 范围。
3. 没进入白名单的能力，不代表永远不做，只代表不在 1.4 一期强求。

## 2. 一期目标重述

结合前面文档，当前建议把 1.4 一期目标定义为：

1. 提供一套独立的本地尺寸工具，用于编辑、验证和发布尺寸方案。
2. 提供标准化、低门槛的标定流程。
3. 让 JetCheck 客户端可以引用已发布方案进行正式运行。
4. 让正式尺寸结果进入现有 JetCheck 检测记录体系。

## 3. 一期白名单总览

建议把一期白名单分成两侧理解：

1. 尺寸工具侧
2. JetCheck 客户端侧

## 4. 尺寸工具侧白名单

### 4.1 工程管理

建议纳入一期：

1. 新建工程
2. 打开工程
3. 最近工程列表
4. 工程基础信息维护
5. 单机本地工程保存

建议不纳入一期：

1. 多人协作
2. 云端工程管理
3. 复杂工程权限控制

### 4.2 图像输入与样本管理

建议纳入一期：

1. 本地图像导入
2. 标定图 / 参考图 / 验证图的基础分类
3. 图像基础信息展示

建议不纳入一期：

1. 尺寸工具内完整相机管理
2. 尺寸工具内直接做复杂实时取流
3. 大规模图像库管理

当前建议：

- 一期先以本地导图为主。
- 如需和 JetCheck 联动，优先考虑后续补“从客户端样本导入”。

### 4.3 标定能力

建议纳入一期：

1. 标准标定板类型选择
2. 内置标定算子执行
3. 向导式标定流程
4. 标定结果展示与保存
5. 标定配置与标定记录追溯
6. 标定适用条件配置

建议不纳入一期：

1. 开放式标定算法参数编排
2. 用户自定义标定板类型
3. 复杂多标定自动切换

### 4.4 第一批标准标定板建议

这里先给出“建议收敛方式”，不把它当最终拍板结论。

建议一期优先直接收敛成：

1. 2 到 3 种标准标定板类型
2. 可以是同样式不同尺寸，也可以是不同样式但仍保持低复杂度

当前建议原因：

1. 先把产品流程跑通，比一开始支持太多板型更重要。
2. 2 到 3 种类型已经能覆盖更多安装条件，整体适用性更好。
3. 一期仍应坚持“标准化能力”，而不是做“开放平台”。

当前待你后续重点审查：

1. 第一批 2 到 3 种板型到底采用哪些尺寸或样式
2. 是否优先采用“同识别逻辑、不同规格”的收敛方式

### 4.5 找正能力

建议纳入一期：

1. 一种主找正方式
2. 搜索 ROI 定义
3. 参考特征定义
4. 允许偏差范围配置
5. 找正结果预览

建议不纳入一期：

1. 多种复杂找正模式并存
2. 高自由度专家级找正参数配置
3. 复杂多目标并行找正

当前建议：

- 一期先把“找正”产品化，而不是铺太多技术分支。

### 4.6 测量项类型

建议纳入一期的高频测量类型：

1. 点到点距离
2. 边到边距离
3. 线到线距离
4. 长度 / 宽度类尺寸
5. 圆 / 孔径类尺寸

建议先不纳入一期：

1. 角度
2. 多段组合尺寸
3. 复杂几何关系
4. 用户自定义测量链

当前建议原则：

- 一期优先覆盖高频静态 2D 测量需求。

### 4.7 测量结果定义

建议纳入一期：

1. 实测值
2. 单位
3. 标称值字段
4. 可选公差字段
5. 可选超差结果字段
6. 叠加图层摘要

当前明确：

1. 测量值输出是一期确定范围。
2. 业务综合结果不在一期范围。
3. 超差判定暂列为“强建议纳入，但待最终确认”。

### 4.8 验证能力

建议纳入一期：

1. 单次验证
2. 轻量批量验证
3. 叠加结果查看
4. 验证任务结果摘要

建议先不纳入一期：

1. 完整统计分析报表
2. 复杂误差分布图
3. 验证报告导出体系

当前建议：

- 没有验证能力，尺寸工具就不算闭环，所以这块必须进一期。

### 4.9 发布能力

建议纳入一期：

1. 发布前检查
2. 尺寸方案
3. 发布版本
4. 方案与标定配置关联
5. 发布到 JetCheck 可消费目录

建议不纳入一期：

1. 复杂版本治理
2. 审批流
3. 多环境发布

## 5. JetCheck 客户端侧白名单

### 5.1 方案获取与引用

建议纳入一期：

1. 客户端可获取已发布尺寸方案目录
2. 运行工具可引用尺寸方案
3. 发布版本可识别、可切换或至少可见

建议不纳入一期：

1. 复杂方案市场
2. 大规模方案分发治理

### 5.2 运行前校验

建议纳入一期：

1. 基础资源可用性校验
2. 发布版本依赖条件校验
3. 标定适用条件校验
4. 找正可恢复性判断
5. 安装异常判断
6. 对应处理动作：
   - 自动恢复
   - 阻断并引导重标定
   - 阻断并提示安装异常

建议不纳入一期：

1. 过于复杂的模糊容错策略
2. 高度可配置的校验策略引擎

### 5.3 正式运行

建议纳入一期：

1. 使用发布版本正式执行
2. 使用当前生效标定记录
3. 执行找正
4. 输出测量值

当前说明：

- 正式运行仍由 JetCheck 承接，这一点是一期开工前就应视为已固定前提。

### 5.4 记录与追溯

建议纳入一期：

1. 记录发布版本
2. 记录标定配置和标定记录
3. 记录相机 / 参数组快照
4. 记录测量项结果列表
5. 记录可选超差结果
6. 记录运行前校验结果

建议不纳入一期：

1. 独立于 JetCheck 的第二套尺寸追溯系统

## 6. 一期明确不纳入白名单的内容

以下内容建议在 1.4 一期明确排除：

1. 平台端训练框架扩展为尺寸工具
2. 云端训练闭环
3. 尺寸工具内完整相机管理
4. 专家级开放算法编排
5. 多人协作工程系统
6. 复杂统计分析平台
7. 业务综合结果判定系统
8. 超复杂多相机联合测量

## 7. 当前最值得你重点审查的白名单项

如果你要做一遍初步审查，我建议你重点盯住下面 6 个点：

1. 一期是否真的收敛成 2 到 3 种标准标定板类型
2. 一期是否真的只做 1 种主找正方式
3. 第一批测量项类型是否收得够窄
4. 轻量批量验证是否一定要进一期
5. 超差判定是否要进一期
6. 客户端侧方案引用和校验范围是否定义得足够保守

## 8. 当前我的建议

如果从“做得稳、做得成”出发，我建议一期按下面这条线走：

1. 标定板类型收窄
2. 找正方式收窄
3. 测量项类型收窄
4. 验证能力保留
5. 发布能力保留
6. 客户端校验和追溯保留

换句话说，一期更像是在做：

- 一个能真正落地的尺寸检测最小产品

而不是：

- 一个面面俱到的专业视觉测量平台
