# JetCheck 1.4 尺寸工具与客户端整合方案 v0.1

本文档基于当前 1.4 相关草案、现有 demo 和本轮讨论结果，整理尺寸方向在 1.4 的整体方案。

重点回答 4 个问题：

1. 尺寸工具在 1.4 里具体做什么
2. JetCheck 客户端基于尺寸类型要改什么
3. 两端如何通过发布产物和运行前校验衔接
4. 重新标定在客户端里如何落地

## 1. 方案定位

1. 1.4 采用 `独立尺寸工具 + JetCheck 客户端融合运行` 的路线。
2. 尺寸工具负责编辑、验证、发布。
3. JetCheck 客户端负责检测工具配置、正式运行、运行前校验、轻量重标定、记录追溯。
4. 尺寸能力在客户端里不单独做一套新搭建流，而是融入现有三步搭建页，作为第三步里的 `尺寸类型` 引用对象。

## 2. 一期目标

1. 尺寸工具可完成工程管理、标定、模版图配置、测量项配置、验证、发布。
2. 客户端第三步可引用尺寸类型对象。
3. 客户端正式运行前可执行尺寸专属运行前校验。
4. 校验失败时，客户端可发起轻量重新标定。
5. 正式运行结果进入现有 JetCheck 检测记录体系。

## 3. 两端职责边界

### 3.1 尺寸工具

尺寸工具负责：

1. 新建和管理本地工程
2. 标定
3. 模版图配置
4. ROI 配置
5. 测量项配置
6. 验证
7. 发布

尺寸工具不负责：

1. 客户端正式运行
2. 客户端相机在线管理
3. 现场运行记录查看
4. 现场异常阻断后的正式处理链路

### 3.2 JetCheck 客户端

客户端负责：

1. 在现有检测工具三步搭建页中引用尺寸类型对象
2. 正式运行前做运行前校验
3. 校验失败时阻断运行
4. 提供轻量重新标定入口
5. 记录正式运行结果、运行前校验结果和实际使用的标定记录

客户端不负责：

1. 编辑尺寸工程
2. 配置 ROI
3. 配置测量项
4. 配置验证集
5. 发布尺寸方案

## 4. 尺寸工具方案

### 4.1 入口与窗口结构

尺寸工具采用 Windows 桌面端应用形态，包含 3 类窗口：

1. 启动窗口
   - 小尺寸登录窗口
   - 输入账号、密码后进入工程管理
2. 工程管理窗口
   - 显示全部工程
   - 支持搜索、新建、导入、设置
   - 状态只保留 `草稿` 和 `已发布`
3. 工程配置窗口
   - 打开某个工程后进入
   - 使用向导式步骤配置工程

### 4.2 工程管理

工程管理页面承载：

1. 工程列表
2. 搜索
3. 新建工程
4. 导入工程
5. 设置入口

工程列表字段收敛为：

1. 工程名称
2. 备注
3. 状态
4. 最近修改时间

设置弹窗展示：

1. 工程所在文件夹
2. 发布输出目录
3. 自动保存开关

### 4.3 工程配置主流程

工程配置窗口采用向导式步骤，当前收敛为 5 步：

1. 标定
2. 模版图配置
3. 测量项配置
4. 验证
5. 发布

其中：

1. 标定允许跳过
2. 验证允许跳过
3. 公差不再单独成步，合并到测量类工具参数中

### 4.4 标定

标定步骤用于建立像素与实际尺寸的换算关系。

界面结构：

1. 左侧是标定图区
   - 上传区和画布区合一
   - 未上传时显示导入入口
   - 上传后显示标定图
2. 右侧是控制区
   - 标定模板
   - 自动标定
   - 像素尺寸

一期先支持：

1. 选择标准标定模板
2. 导入标定图
3. 调用内置算子执行自动标定
4. 输出像素尺寸

### 4.5 模版图配置

第二步改为模版图配置。

界面承载：

1. 上传一张模版图
2. 在模版图上配置 2 个 ROI

一期先把 ROI 做成必做：

1. 检测对象 ROI
2. 检测对象可能出现的区域范围 ROI

### 4.6 测量项配置

第三步是测量项配置。

界面结构：

1. 左侧是工具面板
   - 用 tab 分隔类别
   - 类别按 `元素 / 构造 / 测量`
   - 工具用方形 icon 形式展示
2. 中间是模版图绘制区域
   - 选中工具后直接在图上绘制
3. 右侧是当前测量项参数区
   - 一期只保留测量项名称
   - 测量类工具可选填写上下限

一期先支持高频工具：

1. 元素类
   - 定位点
   - 边缘线
   - 圆特征
2. 构造类
   - 中心轴
   - 中分线
   - 基准框
3. 测量类
   - 长度
   - 角度
   - 直径

### 4.7 验证

验证步骤用于快速检查当前工程配置在样本图上的输出结果。

界面结构：

1. 上传若干验证图
2. 逐张执行当前工程的 ROI 和测量项
3. 左侧结果列表给出 `OK / NG / 异常`
4. 中间图像显示 ROI、测量线和长度值

一期不做：

1. 统计图
2. 复杂验证报表
3. 独立测量结果表

### 4.8 发布

发布页只做发布前检查和发布动作。

页面职责：

1. 校对前几个步骤是否完成
2. 标明必完成项是否完成
3. 满足条件后允许点击发布

发布检查至少包含：

1. 工程基础信息
2. ROI 已配置
3. 测量项已配置
4. 标定是否已补齐
5. 验证是否已执行

其中：

1. 标定可做成强提醒或必填项，取决于最终发布规则
2. 验证允许跳过，但发布页必须明确显示

## 5. 客户端基于尺寸类型的改动

### 5.1 三步搭建页

客户端继续沿用现有三步搭建页：

1. 图像获取
2. 图像处理
3. 图像检测

尺寸能力不新开第四步，而是进入第三步 `图像检测`。

第三步调整为：

1. 检测实例名称
2. 关联输入目标
3. 引用对象

引用对象统一处理为三类：

1. 分类模型
2. 缺陷模型
3. 尺寸类型对象

客户端心智保持统一：

1. 用户仍然是在第三步新增一个检测实例
2. 只是该实例引用的对象可能是尺寸包
3. 是否需要尺寸专属逻辑，在运行时再分流

### 5.2 尺寸类型对象

客户端侧的尺寸类型对象，本质上是尺寸工具发布后的可运行产物。

客户端配置时挂载在 `detect` 实例下，至少要识别：

1. `sceneType = 尺寸`
2. 发布版本信息
3. 依赖的标定配置

第三步配置页不额外塞运行前校验说明，也不承载重标定入口。

### 5.3 运行页

客户端运行页继续沿用原有运行页结构。

尺寸类型运行时需要新增：

1. 发起检测前先执行运行前校验
2. 校验失败则阻断正式检测
3. 校验通过后再进入正式检测

运行页不常驻显示大段校验说明。

需要明确的是，运行前校验发生在 `正式运行时`，不是放在第三步配置页里做一次静态检查。

也就是说：

1. 第三步配置页只负责引用尺寸类型对象
2. 真正点击运行时，客户端才根据当前相机、参数组、分辨率和标定记录做校验
3. 校验不通过时直接阻断本次运行

### 5.4 运行前校验

尺寸类型运行前校验至少检查：

1. 当前工具引用的发布版本是否存在
2. 依赖的标定配置是否存在
3. 当前是否存在生效中的标定记录
4. 当前相机是否可用
5. 当前条件是否满足标定配置适用范围

校验结果分为：

1. `PASS`
2. `PASS_WITH_RECOVERY`
3. `BLOCK_RECALIBRATION_REQUIRED`
4. `BLOCK_INSTALLATION_ABNORMAL`

一期客户端优先落地：

1. `PASS`
2. `BLOCK_RECALIBRATION_REQUIRED`

当前口径还需要把两类问题分开：

1. 当前画面或运行条件与建模时前提明显不一致
   - 这是运行前校验要拦住的问题
   - 例如相机、参数组、分辨率、安装状态明显不匹配
   - 这类情况不能简单认为重标定后一定可恢复
2. 用户已知相机存在轻微偏移，或希望做周期性维护重标定
   - 这是主动重新标定入口要解决的问题
   - 即使当前校验能通过，用户也可以手动发起重新标定

### 5.5 校验失败后的重新标定

校验失败时，客户端弹出失败提醒，不在运行页顶部常驻长文案。

失败提醒后可进入轻量重新标定页。

此外，客户端还应提供 `主动重新标定` 入口，满足下面两类场景：

1. 用户知道相机有轻微偏移，希望先重标定再运行
2. 周期性维护时，需要对当前尺寸检测相机重新补齐标定记录

这个入口放在运行页的 `相机信息` 中，且只对当前正在参与尺寸检测的相机显示。

重新标定页参考尺寸工具第一步“标定”的内容，采用简化版结构：

1. 左侧标定图区
   - 直接显示本地示例标定图
   - 图上不展示识别框和结果框
2. 右侧控制区
   - 标定模板
   - 自动标定按钮
   - 像素尺寸

页面文案收敛为：

1. 只保留 `请将标定板放入当前相机画面`

不再显示：

1. 当前相机
2. 当前对象
3. 常驻解释性长文案

### 5.6 重新标定失败

重新标定不能默认只有成功路径。

一期至少覆盖设备问题导致的失败态：

1. 当前相机离线
2. 当前设备离线
3. 当前输入源不是相机
4. 当前没有可用标定图
5. 标定板未被正确识别

失败时：

1. 自动标定不输出像素尺寸
2. 页面显示明确错误提示
3. 不允许完成并返回

失败提示需要直接告诉用户下一步动作。

建议至少分两类表达：

1. 设备类失败
   - 例如客户端离线、相机离线、输入源异常
   - 提示 `请检查设备连接并调整设备后重试`
2. 识别类失败
   - 例如标定板未完整进入画面、姿态不对、未识别成功
   - 提示 `请调整设备或标定板位置后重试`

为了可演示性，客户端 demo 可以保留一次失败后再成功的流程，用来体现失败引导和重新尝试。

### 5.7 检测记录

客户端记录体系继续统一，但尺寸类型需要补字段。

一期建议补充：

1. 实际使用的发布版本
2. 实际使用的标定记录
3. 运行前校验结果
4. 测量项结果
5. ROI 和测量线叠加结果

## 6. 两端衔接方式

### 6.1 发布产物

尺寸工具发布后至少形成：

1. 尺寸方案
2. 发布版本
3. 标定配置引用关系

客户端消费的是：

1. 发布版本
2. 标定配置
3. 当前生效标定记录

### 6.2 重标定与重发布的边界

这是 1.4 里非常关键的一条边界：

1. 标定失效，不等于尺寸方案失效
2. 重标定默认只更新标定记录
3. 不默认要求重新发布尺寸方案
4. 只有 ROI、测量项、验证结论发生变化时，才需要回到尺寸工具重新验证并重发

## 7. 一期交付内容

### 7.1 尺寸工具

1. Windows 桌面端基础窗口
2. 启动窗口
3. 工程管理窗口
4. 工程配置向导
5. 标定
6. 模版图配置
7. 测量项配置
8. 验证
9. 发布

### 7.2 客户端

1. 第三步支持尺寸类型对象
2. 运行前校验
3. 校验失败阻断
4. 轻量重新标定页
5. 检测记录补充尺寸结果字段

## 8. 当前不做的内容

1. 尺寸工具内完整相机管理
2. 尺寸工具内复杂实时取流
3. 几何公差完整体系
4. 复杂验证报表
5. 多环境发布
6. 云端工程协同

## 9. 当前方案结论

1. 尺寸工具负责把工程从草稿做到可发布。
2. 客户端继续用现有三步搭建页承接尺寸类型。
3. 尺寸类型在客户端里不是新流程，而是第三步的一种引用对象。
4. 真正把尺寸能力和客户端拉开差异的，是运行前校验和重新标定。
5. 1.4 一期优先把 `工程编辑 -> 发布 -> 客户端引用 -> 校验 -> 运行 -> 记录` 这条最小闭环打通。
