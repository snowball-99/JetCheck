# JetCheck 1.4 尺寸工具交互设计草案 v0.1

本文档用于收敛 1.4 尺寸工具在当前阶段的交互设计思路。

当前版本重点回答下面几个问题：

1. 尺寸工具为什么按桌面端应用来设计
2. 工程管理和核心配置步骤各自放在哪里
3. 核心导航采用什么形式
4. 哪些步骤允许先跳过，哪些约束后置到发布前检查

## 1. 当前交互定位

当前对尺寸工具的交互定位明确为：

1. 它不是 JetCheck 客户端里的一个普通配置页。
2. 它是一款独立的桌面端尺寸工程编辑器。
3. 它主要服务质量工程师、工艺工程师、维护调机人员。
4. 它负责编辑态、验证态、发布态，不承接正式产线运行态。

当前阶段不把重点放在“状态表达是否完整”上，而优先保证：

1. 用户能创建工程
2. 用户能完成标定、ROI、测量项、公差、验证、发布等核心配置
3. 用户能在步骤之间自由切换，不被过重流程限制打断

## 2. 为什么按桌面端应用设计

尺寸工具按独立桌面端应用来设计。

原因：

1. 尺寸工具的核心对象是 `工程`，不是一次性表单。
2. 用户的典型动作是“新建工程、打开工程、保存工程、反复修改工程”，而不是“临时打开网页做一次配置”。
3. 一期默认以本地图像导入、本地工程保存为主，更符合单机离线编辑场景。
4. 强工程属性的标定、ROI、测量配置，天然更接近“打开一个桌面编辑器”，而不是“访问一个后台系统页面”。

## 3. 工程管理和步骤流程如何分层

交互分成两层：

1. `工程管理层`
2. `工程内工作流层`

### 3.1 工程管理层

工程管理不放进核心步骤 Tab 中，而放在流程外层。

工程管理层包含：

1. 新建工程
2. 打开工程
3. 最近工程
4. 导入工程
5. 工程基础信息编辑
6. 保存 / 另存为

这一层解决：

1. 我当前在编辑哪个工程
2. 这个工程是否已保存
3. 我要不要切换或打开另一个工程

### 3.2 工程内工作流层

当用户进入某个工程后，再进入核心配置流程。

这一层只承载和当前工程配置直接相关的步骤：

1. 标定
2. ROI
3. 测量项
4. 公差
5. 验证
6. 发布

这一层解决：

1. 当前工程还需要配置什么
2. 我现在正在做哪一步
3. 下一步我想去哪里继续配置

## 4. 导航形式

工程内主导航采用 `向导式 Tab`。

形式如下：

1. 顶部或内容区上方放一排横向 Tab
2. Tab 顺序固定为：
   - 标定
   - ROI
   - 测量项
   - 公差
   - 验证
   - 发布
3. 用户可直接点击任意 Tab 进入对应步骤
4. 不在步骤切换时做过重阻断

不采用下面这些形式：

1. 只能顺序点击下一步的强向导
2. 左侧很重的树状导航
3. 大量状态解释和流程说明并列堆在主导航里

原因：

1. 当前阶段最重要的是把核心配置链路走通。
2. 工程师在真实使用中，往往需要来回切换步骤反复调整。
3. 如果把前置约束全压在 Tab 切换上，交互会变得僵硬。

## 5. 流程约束方式

流程约束采用 `弱顺序、强发布检查`。

含义：

1. 步骤顺序仍然保留，帮助用户理解推荐流程。
2. 但步骤之间允许自由切换，不强制锁死。
3. 真正的约束尽量后置到发布前检查，而不是前置到步骤切换。

这意味着：

1. 可以先配测量项，再回头补标定。
2. 可以先做 ROI，再调整公差。
3. 可以不做验证，直接尝试发布。

但在最终发布前，系统仍应做一次统一检查。

## 6. 对“可跳过步骤”的当前口径

### 6.1 标定

标定的口径如下：

1. 标定可以后做，不要求用户一进入工程就先完成。
2. 用户可以先完成 ROI、测量项等配置，再回头补标定。

原因：

1. 有些场景下，用户会先把测量关系和图像结构理顺。
2. 标定更多决定真实尺寸换算，不一定必须成为第一个操作动作。

但需要明确：

1. 标定是可以后置，不是永久不需要。
2. 当发布版本需要输出真实物理尺寸时，标定仍然是关键前提。

### 6.2 验证

验证的口径如下：

1. 验证允许跳过。
2. 用户可以不做验证，直接发版。

原因：

1. 当前阶段应先允许最短路径完成配置闭环。
2. 有些用户会先发布试运行版本，再补验证样本。

“未验证”不作为禁止发布的硬条件。

### 6.3 公差

公差的口径如下：

1. 公差独立成一个步骤是合理的。
2. 公差属于测量结果的补充定义，不要求每个测量项都强制配置。

这一步的重点应是：

1. 给测量项补标称值
2. 配上下限
3. 决定是否输出 OK / NG

## 7. 界面骨架

进入工程后的工作台，采用下面的基础布局：

1. `顶部工具栏`
2. `向导式 Tab`
3. `中央主工作区`
4. `右侧轻属性区`

### 7.1 顶部工具栏

顶部工具栏包含：

1. 工程名称
2. 工程基础信息入口
3. 保存
4. 另存为
5. 导入工程
6. 发布入口

### 7.2 向导式 Tab

向导式 Tab 承载工程内步骤切换：

1. 标定
2. ROI
3. 测量项
4. 公差
5. 验证
6. 发布

### 7.3 中央主工作区

这是主操作区，始终围绕图像与叠加进行。

主工作区保持下面的组织方式：

1. 中间固定保留大图像区
2. 所有步骤都尽量在同一主画布心智上完成
3. 不同步骤只切换当前操作模式和配置面板

### 7.4 右侧轻属性区

右侧属性区保持轻量，不承载太重的“阶段结论”表达。

右侧属性区包含：

1. 当前步骤的配置项
2. 当前选中对象的属性
3. 当前图像或测量项的小摘要

当前阶段不优先承载：

1. 大段结果说明
2. 重状态结论区
3. 复杂发布准备度说明

这些内容可在后续交互完善阶段再补强。

## 8. 各 Tab 内容

### 8.1 标定

标定 Tab 包含：

1. 选择标准标定板类型
2. 导入标定图
3. 执行识别
4. 查看标定叠加效果
5. 保存标定结果

### 8.2 ROI

ROI Tab 包含：

1. 选择参考图
2. 定义 ROI 区域
3. 配置 ROI 内的参考对象
4. 查看 ROI 叠加效果
5. 调整 ROI 范围与位置

### 8.3 测量项

测量项 Tab 包含：

1. 新增测量项
2. 选择测量类型
3. 选择对象或几何关系
4. 在图像上生成叠加元素
5. 编辑测量项名称、单位、精度

### 8.4 公差

公差 Tab 包含：

1. 选择某个测量项
2. 配标称值
3. 配上限和下限
4. 决定是否输出超差结果

### 8.5 验证

验证 Tab 包含：

1. 选择验证图或验证集
2. 发起单次或批量验证
3. 查看测量值与叠加结果
4. 查看异常样本

这一项不作为发布前硬门槛。

### 8.6 发布

发布 Tab 包含：

1. 发布前检查
2. 填写版本信息
3. 生成发布版本
4. 发布到 JetCheck 可消费目录

主要约束集中到发布前检查，而不是分散在步骤切换中。

## 9. 发布前检查口径

虽然当前阶段不强调每一步的重状态表达，但发布前仍保留最基本检查。

发布前至少检查：

1. 是否已有工程基础信息
2. 是否已有至少一个测量项
3. 是否已有可用 ROI 设置
4. 标定是否已补齐到可发布要求
5. 是否跳过验证

检查结果按下面口径处理：

1. `测量项缺失` 应视为不可发布
2. `验证缺失` 允许发布
3. `标定是否必拦截` 需要结合最终“是否允许先发布非真实尺寸版本”再细化

## 10. 当前阶段不优先展开的内容

当前版本先不把精力放在下面这些点上：

1. 每一步的复杂完成态定义
2. 大量结论区、摘要区、状态区设计
3. 复杂的步骤锁定关系
4. 过重的异常引导文案

原因：

1. 当前阶段最重要的是把核心配置交互走顺。
2. 先保证用户真的能完成一次配置，再补状态表达，会更稳。

## 11. 总结

当前对尺寸工具交互的结论可归纳为：

1. 按桌面端应用来设计，核心对象是 `工程`。
2. 工程管理放在流程外层，不放进核心步骤 Tab。
3. 工程内主导航采用可跳转的向导式 Tab。
4. 流程采用“弱顺序、强发布检查”，不在步骤切换时做过重阻断。
5. 标定可后置，验证可跳过，公差可按需补充。
6. 当前阶段先把核心功能步骤走通，再补每一步的状态表达与结果结论设计。
