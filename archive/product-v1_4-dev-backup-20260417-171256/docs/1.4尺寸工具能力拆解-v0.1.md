# JetCheck 1.4 尺寸工具能力拆解 v0.1

本文档用于进一步明确“尺寸工具本身到底要具备哪些能力”。

这里讨论的对象是尺寸测量工具本身，而不是 JetCheck 运行页。默认前提仍然是：

1. 尺寸工具负责编辑态、验证态、发布态
2. JetCheck 负责正式运行态、记录态和现场集成

## 1. 先明确尺寸工具的产品定位

尺寸工具不应被理解成一个“给现有检测工具加几个尺寸字段的小功能”，而更接近一款本地化的尺寸工程编辑器。

它的核心目标不是直接跑产线，而是帮助用户完成以下事情：

1. 创建尺寸工程
2. 配置标定和找正基准
3. 配置测量项和可选公差
4. 用样本图做验证
5. 产出一个可被 JetCheck 客户端消费的尺寸方案发布版本 / 运行包

如果这个定位成立，那么尺寸工具的能力应该优先围绕“工程编辑闭环”展开，而不是围绕“现场运行闭环”展开。

## 2. 尺寸工具的能力结构建议

我建议把尺寸工具拆成 8 个能力域来思考。

### 2.1 工程管理

这是工具的最外层能力，决定用户怎么组织自己的测量方案。

建议至少包含：

1. 新建工程
2. 打开工程
3. 最近工程列表
4. 工程基础信息
   - 工程名称
   - 适用产品 / 型号
   - 适用工位
   - 备注说明
5. 工程内资源组织
   - 标定配置
   - 样本图
   - 测量项
   - 验证集
   - 发布版本

为什么这块重要：

1. 尺寸工具不是一次性页面，而是会反复打开、修改、验证、发布。
2. 后续尺寸方案同步给客户端时，也需要一个清晰的工程来源。

一期建议：

- 支持单机本地工程管理即可，不急着做多人协作。

### 2.2 图像与样本管理

尺寸工具必须有明确的“输入图像管理”能力，否则标定、找正、测量、验证都会散掉。

建议至少包含：

1. 导入本地图像
2. 导入标定图像
3. 导入参考图
4. 导入验证图像
5. 对图像做基础分类
   - 标定图
   - 参考图
   - 验证图
6. 查看图像元信息
   - 分辨率
   - 来源相机
   - 参数组
   - 导入时间

建议和 JetCheck 的关系：

- JetCheck 负责相机资源和运行侧真实取图。
- 尺寸工具负责消费样本图，不重复做完整相机管理。

一期建议：

- 先以本地导入为主。
- 如果要和 JetCheck 联动，可后续补“从客户端样本中拉取”。

### 2.3 标定能力

标定是尺寸工具的基础能力，不是附属功能。

这里建议的产品方向已经比较清晰：

1. 系统预置若干种标准标定板类型
2. 每种标定板绑定内置标定算子
3. 用户主要通过向导式流程完成标定

建议标定能力至少包含：

1. 选择标定板类型
2. 采集或导入标定图像
3. 自动识别标定板
4. 显示识别结果与有效性
5. 保存当前标定结果
6. 绑定适用条件
   - 相机
   - 参数组
   - 分辨率
   - 可选镜头 / 工位条件
7. 保留标定记录用于追溯

这里要特别强调：

- 标定解决的是“像素到物理尺寸的换算关系”。
- 它不是简单写在相机上的固定属性，而是和具体成像条件相关。

一期建议：

- 先支持少量标准标定板。
- 先支持一个尺寸工程绑定一个主标定配置。
- 不急着做复杂的多标定自动切换。
- 用户侧优先感知“当前标定是否有效”，不暴露复杂版本管理。

这里需要补充一个重要边界：

1. 标定不等于万能恢复能力。
2. 标定主要解决像素到物理尺寸的换算关系。
3. 如果现场变化已经破坏了原本的成像几何条件，不能简单认为“重新标定一下就一定能继续跑”。

基于这一点，建议把现场变化分成三类：

1. 找正可解决
   - 例如工件在画面中的位置变化、小范围平移或旋转
2. 重新标定可解决
   - 例如工作距离轻微变化、比例关系变化，但相机姿态仍在允许范围内
3. 安装异常
   - 例如相机角度变化明显、整体倾斜或偏移过大、透视关系明显失真

一期建议采用保守策略：

- 能自动恢复的自动恢复
- 需要重标定的引导重标定
- 超出允许范围的直接阻断

### 2.4 找正能力

这部分很容易被忽略，但实际上和标定同样关键。

标定解决“比例换算”，找正解决“每次工件位置变化后，怎么还能测同一个位置”。

建议至少包含：

1. 定义搜索 ROI
2. 定义参考特征
   - 边
   - 角点
   - 圆
   - 基准线
3. 在 ROI 内执行参考匹配
4. 支持一定范围的平移 / 旋转恢复
5. 在样本图上验证找正稳定性
6. 显示找正后的测量叠加效果

为什么这是基础能力：

1. 只做标定不做找正，物体一旦偏移，测量位置就不稳定。
2. 现场尺寸检测通常都需要某种形式的找正、匹配或基准定位。

一期建议：

- 先支持一种主找正方式，避免一开始做太多。
- 产品上优先选择最稳定、最好理解的一种方式。

这里和标定的关系需要明确：

1. 标定回答的是“像素怎么换算成真实尺寸”。
2. 找正回答的是“这次图像里应该去哪里量、当前偏了多少”。
3. 一般情况下，工件位置变化优先由找正处理，而不是触发重新标定。
4. 只有当成像几何条件变化到超出原标定前提时，才进入重标定或阻断逻辑。

### 2.5 测量项配置能力

这是尺寸工具最核心的“业务能力层”。

建议把测量项理解为“面向用户的测量结果定义”，而不是直接暴露底层算法细节。

每个测量项建议至少包含：

1. 测量项名称
2. 测量类型
3. 关联的几何基元或检测区域
4. 输出单位
5. 显示精度
6. 可选公差范围
7. 可视化叠加样式
8. 结果说明

测量类型建议不要一开始做得过宽，而是围绕高频场景收敛。

一期建议优先支持：

1. 点到点距离
2. 边到边距离
3. 线到线距离
4. 圆 / 孔径类尺寸
5. 长度 / 宽度类尺寸

可后续再考虑：

1. 角度
2. 多段组合尺寸
3. 复杂几何关系
4. 用户自定义测量链

一个关键原则：

- 产品上尽量让用户在“配置测量项”，而不是“拼算法节点”。

### 2.6 公差与结果定义

尺寸工具虽然不负责 JetCheck 的业务综合结果，但仍然需要定义“测量结果自身的结构”。

建议每个测量项的输出至少支持：

1. 实测值
2. 单位
3. 标称值
4. 上下公差
5. 是否超差
6. 原始叠加图层

这里建议把结果分成两层：

1. 测量结果层
   - 属于尺寸工具本身
2. 业务结果层
   - 属于 JetCheck 现有业务判定体系

这样做的好处是：

1. 责任边界清晰
2. 后续即使不在一期做业务综合判定，记录结构也不会失真
3. JetCheck 可以继续用自己的业务口径做最终整合

一期建议：

- 测量值一定要有。
- 超差判定建议尽量纳入，但如果来不及，可以先把数据结构预留好。

### 2.7 验证能力

验证能力是尺寸工具区别于“静态配置页”的关键。

如果没有验证能力，用户很难确认尺寸方案到底能不能用。

我建议至少分成两层：

#### 单次验证

能力包括：

1. 选一张图试跑
2. 查看找正结果
3. 查看测量叠加线 / 点 / 框
4. 查看测量值输出
5. 查看可选超差结果

#### 小批量验证

能力包括：

1. 选一组验证图批量执行
2. 查看每张图的测量结果
3. 快速看哪些图表现异常
4. 快速切换查看叠加结果

如果资源允许，可进一步增加：

1. 验证通过率
2. 单测量项统计分布
3. 误差摘要
4. 发布前检查清单

一期建议：

- 单次验证必须有。
- 小批量验证强烈建议有，但可以做轻量版。
- 统计报表不是一期硬要求。

### 2.8 发布与打通能力

尺寸工具最终不能停留在“本地能配出来”，必须能被 JetCheck 消费。

建议至少包含：

1. 发布前检查
   - 是否已有标定
   - 是否已有找正基准
   - 是否至少有一个测量项
   - 是否做过基本验证
2. 发布产物生成
3. 发布版本号
4. 记录发布时间、来源工程、关联的标定配置
5. 提供给 JetCheck 客户端可识别的发布包

这里需要尽早明确一个产品对象：

- 客户端看到的到底是什么：
  - 尺寸工程
  - 尺寸方案
  - 发布版本
  - 可运行包

我目前更建议客户端侧看到的是“尺寸方案 / 发布版本”，而不是编辑态工程本身。

还建议在发布产物中明确带出：

1. 依赖的标定配置 / 当前标定记录
2. 允许的运行条件
3. 超出条件时的处理策略
   - 自动恢复
   - 需重标定
   - 阻断

## 3. 这些能力中，哪些是一期必须做的

如果按 1.4 一期做“最小但完整闭环”，我建议必须具备：

1. 本地工程管理
2. 样本图导入
3. 标准标定板向导式标定
4. 一种稳定的找正方式
5. 少量高频测量项类型
6. 测量值输出
7. 单次验证
8. 轻量小批量验证
9. 发布尺寸方案 / 发布版本
10. JetCheck 客户端消费已发布方案
11. JetCheck 记录中展示尺寸结果
12. JetCheck 运行前对标定条件和安装异常做基础校验

## 4. 一期不建议做太满的能力

以下能力可以先明确“不在 1.4 一期强求”：

1. 开放式自定义标定算法配置
2. 太多种标定板类型
3. 太多种几何测量算子
4. 脚本化流程编排
5. 完整专家级统计分析
6. 多人协同编辑
7. 云端协同训练闭环
8. 超复杂多相机联合测量

## 5. 当前最值得继续深挖的 5 个能力点

如果我们下一轮继续往深里收，我建议优先深挖这 5 个点：

1. 标定能力
   - 标定板类型、标定流程、标定结果结构
2. 找正能力
   - 用什么方式保证工件偏移后仍能稳定测量
3. 测量项模型
   - 一期支持哪些测量类型
4. 验证能力
   - 单次验证和批量验证做到什么程度
5. 发布与同步
   - 尺寸方案如何从尺寸工具流向 JetCheck 客户端

## 6. 当前我的总体判断

如果后续不把能力结构提前拆清楚，尺寸工具很容易变成一个“能画线、能出数，但对象关系混乱”的半成品。

真正决定 1.4 成败的，不只是算法本身，而是这几个能力能不能闭环：

1. 标定闭环
2. 找正闭环
3. 测量配置闭环
4. 验证闭环
5. 发布到 JetCheck 的闭环

我当前建议的方向是：

- 用产品化方式收敛标定
- 用有限测量类型覆盖高频场景
- 用验证能力保证尺寸方案可用
- 用发布机制打通 JetCheck

这样 1.4 才像一个真正可落地的尺寸检测方案，而不是一版概念 demo。
