# JetCheck 1.4 运行前校验规则草案 v0.1

本文档用于定义 JetCheck 客户端在正式执行尺寸检测前，应如何做运行前校验。

目标是把下面几件事变成可执行规则：

1. 哪些条件必须校验
2. 哪些问题允许自动恢复
3. 哪些问题需要重标定
4. 哪些问题必须阻断

## 1. 设计目标

1. 保证尺寸结果可信，不在明显不可信的条件下继续运行
2. 对轻微、可恢复问题尽量自动恢复
3. 对需要重标定的情况给出清晰引导
4. 对超出允许范围的情况及时阻断
5. 让记录里能追溯本次运行前校验结果

## 2. 当前校验策略总原则

建议按以下优先级处理：

1. 先校验“是否满足基础运行条件”
2. 再校验“是否满足发布版本依赖条件”
3. 再判断“是否属于可自动恢复范围”
4. 最后决定：
   - 放行
   - 放行但记录自动恢复
   - 阻断并引导重标定
   - 阻断并提示安装异常

这里要特别区分两件事：

1. 运行前校验
   - 解决“当前条件下能不能安全运行”
   - 发生在用户点击运行时
2. 主动重新标定
   - 解决“虽然当前可能还能跑，但用户希望更新像素尺寸换算关系”
   - 入口可以独立存在，不依赖校验失败后才触发

## 3. 校验输入

运行前校验建议至少读取以下输入。

### 3.1 当前运行输入

1. 当前工具引用的发布版本
2. 发布版本依赖的标定配置
3. 当前相机
4. 当前参数组
5. 当前分辨率
6. 当前取图模式

### 3.2 发布版本侧约束

1. 发布版本允许的相机范围
2. 发布版本允许的参数组范围
3. 发布版本依赖的标定配置 ID
4. 发布版本要求的运行条件

### 3.3 标定侧约束

1. 标定配置的适用条件
2. 当前生效的标定记录
3. 标定记录的质量状态

## 4. 建议的校验分层

我建议把运行前校验拆成 5 层。

### 4.1 第一层：基础资源可用性校验

这一层主要判断“能不能开始跑”。

建议校验项：

1. 相机是否在线
2. 相机是否可访问
3. 参数组是否存在
4. 当前发布版本是否可用
5. 依赖的标定配置是否存在
6. 当前是否存在生效中的标定记录

处理建议：

1. 任一项失败，直接阻断
2. 这类问题不属于自动恢复范围

### 4.2 第二层：发布版本依赖条件校验

这一层主要判断“当前运行资源是不是发布版本允许的对象”。

建议校验项：

1. 当前相机 ID 是否在发布版本允许范围内
2. 当前参数组是否在发布版本允许范围内
3. 当前分辨率是否在发布版本允许范围内
4. 当前运行模式是否满足发布版本要求

处理建议：

1. 不匹配时默认阻断
2. 不建议一期用模糊兼容策略放过

### 4.3 第三层：标定适用条件校验

这一层主要判断“当前条件是否还在标定配置的适用范围内”。

建议校验项：

1. 当前相机是否匹配标定配置
2. 当前参数组是否匹配标定配置
3. 当前分辨率是否匹配标定配置
4. 当前工位条件是否匹配标定配置

处理建议：

1. 如果条件明显不匹配，阻断并引导重标定
2. 如果只是边界内的轻微变化，可继续走下一层判断

补充说明：

1. 标定适用条件校验的目的是判断“当前前提是否仍在原标定记录的适用范围内”
2. 这层通过，不代表用户不能主动重新标定
3. 这层失败，也不代表一定只是重标定问题，仍要结合后续安装异常判断

### 4.4 第四层：找正可恢复性判断

这一层主要判断“当前画面偏差是否还能靠找正解决”。

建议判断内容：

1. 参考区域是否可识别
2. 参考特征是否可识别
3. 找正偏移量是否在允许范围内
4. 旋转 / 平移量是否在允许范围内

处理建议：

1. 若在允许范围内：
   - 放行
   - 并记录“已自动恢复”
2. 若超出允许范围：
   - 进入下一层安装异常判断

### 4.5 第五层：安装异常判断

这一层主要判断“当前是否已经超出发布版本和标定的前提”。

建议判定为安装异常的典型情况：

1. 画面角度变化明显
2. 整体倾斜或偏移过大
3. 透视关系明显异常
4. 当前画面中基准特征已失真到不可用

处理建议：

1. 直接阻断
2. 提示先调整相机安装
3. 调整后再重新验证或重标定

## 5. 校验结果分级

建议把运行前校验结果分成 4 级。

### 5.1 `PASS`

说明：

- 所有运行条件满足，允许直接运行。

### 5.2 `PASS_WITH_RECOVERY`

说明：

- 系统已通过找正自动恢复到可运行状态。

建议行为：

1. 允许继续运行
2. 在记录中写入“已自动恢复”
3. 不建议一期在主界面做太重提示，但应可追溯

### 5.3 `BLOCK_RECALIBRATION_REQUIRED`

说明：

- 当前条件已超出原标定适用范围，但仍像是“可通过重标定恢复”的场景。

建议行为：

1. 阻断正式运行
2. 提示使用标准标定板重新标定
3. 不默认要求重新发版
4. 支持进入轻量重新标定窗口

### 5.4 `BLOCK_INSTALLATION_ABNORMAL`

说明：

- 当前条件已经像“安装异常”，不应仅靠重标定兜底。

建议行为：

1. 阻断正式运行
2. 提示先调整安装
3. 调整完成后再重新验证

这一类不应被产品简单包装成“去重标定就行”。

## 6. 一期建议的默认规则

如果一期采用保守策略，我建议默认规则如下：

1. 相机离线：直接阻断
2. 参数组不存在：直接阻断
3. 发布版本依赖对象缺失：直接阻断
4. 发布版本允许范围不匹配：直接阻断
5. 标定配置适用条件不匹配：阻断并引导重标定
6. 找正偏移量在允许范围内：自动恢复并继续运行
7. 找正偏移量超限，但画面还能看清：优先按需重标定处理
8. 画面已明显失真或姿态异常：直接按安装异常阻断

## 7. 重新标定后的处理建议

这是当前校验规则里非常关键的一段。

建议：

1. 当校验结果为 `BLOCK_RECALIBRATION_REQUIRED` 时，不默认要求重新发版
2. 重标定后，在原标定配置下生成新的标定记录
3. 校验通过后，把新记录切换为生效记录
4. 后续运行沿用原发布版本

这条规则的核心是：

- 把“发布版本问题”和“标定问题”分开处理

补充一条主动重新标定规则：

1. 即使运行前校验通过，用户也可以从相机信息入口主动发起重新标定
2. 主动重新标定成功后，同样是在原标定配置下生成新的标定记录
3. 后续运行优先使用新的生效标定记录

## 8. 建议写入记录的校验信息

运行前校验结果建议写入 JetCheck 检测记录。

建议字段：

1. 校验状态
2. 校验时间
3. 校验摘要
4. 是否发生自动恢复
5. 实际使用的标定记录
6. 当前相机 / 参数组 / 分辨率快照

这样后续才能回答：

1. 这次运行是否在标准条件下执行
2. 是否发生过自动恢复
3. 是否使用了更新后的标定记录

## 9. 当前最建议补充到产品交互中的提示

为了让现场更可用，我建议至少要有下面几类提示：

1. 可直接运行
2. 已自动恢复，可继续运行
3. 当前条件需要重标定，暂不能运行
4. 当前安装异常，请先调整相机安装

提示原则建议：

1. 不要只说“失败”
2. 要明确告诉用户下一步该做什么

对于重新标定窗口内的失败提示，建议进一步收敛为两类：

1. 设备类失败：请检查设备连接并调整设备后重试
2. 识别类失败：请调整设备或标定板位置后重试

## 10. 当前仍需继续细化的点

这版规则还需要后续继续细化：

1. 找正允许范围如何定义
2. “需重标定”和“安装异常”的分界线如何产品化表达
3. 主动重新标定入口是否只对尺寸检测链路中的相机展示
4. 自动恢复是否需要被用户显式确认

## 11. 当前建议

如果按现在的方向继续推进，我建议先接受下面这 5 条默认口径：

1. 运行前必须校验
2. 可自动恢复的只限可信范围内
3. 需要重标定时阻断正式运行
4. 超出安装允许范围时直接阻断
5. 校验结果必须进入记录追溯
