# JetCheck 项目定制需求汇总 v1

## 1. 文档目的

本文基于当前已整理的 6 个试点项目需求文档，提炼：

1. 标准产品已经具备的共性能力
2. 各项目之间反复出现的差异化变量
3. 后续方案设计中应优先抽象为配置、流程编排或结果渲染策略的能力
4. 当前跨项目仍待补齐的关键信息

本文不是单项目方案文档，而是面向后续“标准产品 + 项目定制”设计的总表。

## 2. 当前已纳入汇总的项目

1. 天阁压壳螺纹有无攻丝检测
2. 天阁背板缺陷检测
3. 马斯特 X 光加热丝缺陷检测
4. 三洋安全座椅零部件视觉检测
5. 血氧纱布缺陷检测
6. 瑾辰汽车饰件视觉检测

## 3. 当前标准产品可承接的共性能力

结合已有 demo、PRD 基线和项目梳理结果，当前标准产品已经具备或已明显形成方向的能力主要有：

### 3.1 工具配置主链路

- 统一的三段式配置链路：
  - 图像获取
  - 图像处理
  - 图像检测
- 图像获取支持：
  - 相机输入
  - 接口输入
- 图像处理支持：
  - 全图处理
  - 手绘 ROI
  - 模型 ROI
- 图像检测支持：
  - 分类类
  - 缺陷检测类
  - 尺寸/测量类

### 3.2 运行态主框架

- 统一的运行态页面框架：
  - 当前运行状态
  - 每张图结果
  - 当前图像结果
  - 总结果
  - 标签信息
- 支持外部信号驱动的运行逻辑概念。
- 支持多图结果逐步展示和总结果汇总。

### 3.3 记录与追溯主结构

- 已形成三层记录结构：
  - 总记录
  - 图像记录
  - 子图/子结果记录
- 已具备结果查看、ROI/子图查看、基础导出思路。

### 3.4 资源管理能力

- 模型管理
- 相机管理
- 参数组管理
- 存储提醒、阻断、清理

### 3.5 适合作为标准产品保留的能力判断

下列能力在多个项目中重复出现，建议明确作为标准产品长期能力保留：

1. 图像输入源抽象
2. 多图任务抽象
3. 手绘 ROI / 模型 ROI 抽象
4. 多检测项汇总抽象
5. 总结果 / 单图结果 / 子结果三层结果结构
6. 记录查看与结果图查看器

## 4. 六个项目的共性与差异

### 4.1 共性

六个项目有以下明显共性：

1. 使用角色高度一致
   - 主要都是质检操作员
2. 现场大多数为离散工位或半离散工位
   - 人工上料
   - 人工触发
   - 人工下料或人工处理异常
3. 项目最终都落到业务结果判定
   - OK / NG
4. 大多数项目都需要把算法结果转换成业务结果
   - 不是直接使用模型输出
5. 大多数项目都存在异常处理要求
   - 超时
   - 漏图
   - 模型异常
   - 存储不足
6. 大多数项目都希望在 NG 时看到更具体的位置或测量信息

### 4.2 关键差异

差异主要集中在 5 个维度：

1. 触发与流程联动方式不同
2. 图像组织方式不同
3. 检测链路复杂度不同
4. 业务判定逻辑不同
5. 结果展示密度不同

## 5. 跨项目变量总表

### 5.1 触发与工位流程变量

| 维度 | 典型取值 | 涉及项目 |
| --- | --- | --- |
| 触发来源 | PLC 信号 | 天阁压壳 |
| 触发来源 | 踏板 / 实体按钮 | 天阁背板、三洋、血氧纱布、瑾辰 |
| 触发来源 | 文件/接口逐张处理 | 马斯特 X 光 |
| 前序工序联动 | 前一道工序 NG 直接跳过视觉 | 天阁压壳 |
| 过程设备联动 | 旋转气缸 / 机械臂 / 桁架移动 | 天阁背板、血氧纱布、瑾辰 |
| 后续系统联动 | 结果回传 PLC / MES | 天阁压壳、天阁背板 |

结论：

- “如何开始一次检测”不能再当作单一固定逻辑，应抽象为可配置触发器。
- “一次检测中是否存在中途动作联动”也需要抽象，而不是只支持一次触发一次出结果。

### 5.2 图像组织变量

| 维度 | 典型取值 | 涉及项目 |
| --- | --- | --- |
| 每次任务图像数 | 1 张 | 天阁压壳、马斯特 X 光 |
| 每次任务图像数 | 2 张 | 三洋、瑾辰 |
| 每次任务图像数 | 4 张 | 天阁背板 |
| 每次任务图像数 | 16 张 | 血氧纱布 |
| 图像来源 | 相机 | 天阁压壳、天阁背板、三洋、血氧纱布、瑾辰 |
| 图像来源 | 接口/指定位置 | 马斯特 X 光 |
| 单图内容 | 单零件 | 天阁压壳、三洋、血氧纱布、瑾辰、天阁背板 |
| 单图内容 | 多零件混料 | 马斯特 X 光 |

结论：

- “一个任务 = 一张图”不能作为默认前提。
- 标准产品必须支持“任务由 N 张图组成”的抽象，且 N 可变。

### 5.3 图像处理变量

| 维度 | 典型取值 | 涉及项目 |
| --- | --- | --- |
| 无 ROI / 全图 | 背板、三洋 |
| 手绘 ROI | 瑾辰 |
| 模型 ROI | 马斯特 X 光 |
| 分区/裁块型 ROI | 血氧纱布 |
| 可能需要限定主体区域 | 天阁压壳 |

结论：

- ROI 不能只理解为“手动画框”。
- 在项目层面，ROI 至少分成 4 类：
  - 全图
  - 手绘固定 ROI
  - 模型生成 ROI
  - 分区/切块 ROI

### 5.4 检测链路变量

| 维度 | 典型取值 | 涉及项目 |
| --- | --- | --- |
| 单模型单检测项 | 天阁背板 |
| 单图多检测项 | 三洋、瑾辰 |
| 多模型串联 | 马斯特 X 光 |
| 单任务含分类 + 尺寸 | 三洋 |
| 单任务含卡扣 + 焊点 | 瑾辰 |
| 分面/翻面两轮检测 | 天阁背板、血氧纱布 |

结论：

- “一个图像处理实例只接一个检测模型”的理解不够。
- 后续方案需要支持：
  - 一个任务中多个检测项并列
  - 多模型串联
  - 同一图像结果挂多个业务判断项

### 5.5 结果判定变量

| 维度 | 典型取值 | 涉及项目 |
| --- | --- | --- |
| 计数型判定 | 2 个有攻丝且 0 个无攻丝才 OK | 天阁压壳 |
| 任意缺陷即 NG | 天阁背板、马斯特 X 光、血氧纱布 |
| 任意检测项 NG 则总 NG | 三洋、瑾辰 |
| 型号级公差 | 三洋 |
| 可能需要阈值配置 | 天阁背板、马斯特 X 光、血氧纱布 |
| 明确不看阈值 | 天阁压壳、三洋 |

结论：

- 业务判定不能内嵌死在模型逻辑里。
- 应单独抽象“业务判定规则层”，接收算法结果后再输出业务结果。

### 5.6 结果可视化变量

| 维度 | 典型取值 | 涉及项目 |
| --- | --- | --- |
| 只看最终 OK/NG | 多项目基础诉求 |
| 看缺陷位置 | 天阁背板、马斯特 X 光、血氧纱布、瑾辰 |
| 看 ROI 区域位置 | 血氧纱布、瑾辰 |
| 看测量尺寸 | 三洋 |
| 看 OK/NG 数量统计 | 马斯特 X 光 |
| 看按类别/型号统计 | 马斯特 X 光 |

结论：

- 结果展示不能只有一种固定模板。
- 至少要支持：
  - 全图框叠加
  - ROI 级定位
  - 测量值展示
  - 统计摘要展示

## 6. 建议抽象成标准配置的能力

基于以上共性与差异，建议优先抽象为标准产品配置项的能力如下。

### 6.1 任务编排配置

建议把“检测工具”升级为“任务编排”概念，至少可配置：

1. 任务由几张图组成
2. 每张图由什么触发
3. 图与图之间是否有中间动作
4. 是否需要翻面/旋转/移动
5. 某一步异常后是否中断后续步骤

### 6.2 图像输入配置

建议抽象以下配置：

1. 输入类型
   - 相机
   - 接口
   - 指定目录/文件流
2. 输入数量
3. 输入顺序
4. 输入命名和面别规则

### 6.3 ROI 策略配置

建议把 ROI 抽象为独立策略：

1. 全图
2. 手绘固定 ROI
3. 模型 ROI
4. 规则分区 ROI
5. 多次物理采图分区

### 6.4 业务判定配置

建议把业务判定独立配置化，至少支持：

1. 任一 NG 即总 NG
2. 计数满足条件才 OK
3. 型号/件别相关规则
4. 阈值规则
5. 尺寸公差规则
6. 不匹配/异常/待确认等中间状态定义

### 6.5 记录与追溯配置

建议独立配置：

1. 任务级记录字段
2. 图像级记录字段
3. ROI/子结果级记录字段
4. 批次号、产品号、序号生成规则
5. 首检/复检记录保留策略
6. MES 字段映射规则

### 6.6 结果渲染配置

建议抽象可选渲染块：

1. 总结果卡片
2. 图像列表
3. ROI 位置图层
4. 缺陷框图层
5. 测量值图层
6. 统计摘要
7. 件别/型号说明

## 7. 建议抽象成流程编排能力的部分

有些能力不只是简单配置字段，而更像“流程节点编排”，建议后续单独设计。

### 7.1 典型流程节点

1. 等待外部触发
2. 采图
3. 模型 ROI
4. ROI 检测
5. 多检测项汇总
6. 结果回传
7. 中间设备动作
8. 人工复检回路

### 7.2 典型分支逻辑

1. 前序工序 NG 则跳过视觉
2. 某一步超时则中断任务
3. 某图 NG 后是否继续完成剩余图
4. 首检 NG 后是否进入复检回路

## 8. 当前跨项目最值得优先澄清的信息

虽然六个项目都已形成初稿，但跨项目看，仍有 6 类信息经常缺失，而且会直接影响后续方案设计：

### 8.1 触发与回传协议

- 触发方是谁
- 结果回给谁
- 返回哪些状态码

### 8.2 节拍与超时

- 单图超时
- 单任务超时
- 异常后是否中断整批/整件

### 8.3 判定中间态

- 不匹配
- 漏检
- 多检
- 低置信度
- 异常

这些状态到底是：

- 直接 NG
- 算异常
- 进入人工确认
- 仅做过程信息

### 8.4 复检策略

- 是否允许复检
- 复检是否覆盖首检
- MES 取哪条记录

### 8.5 追溯字段

- 批次号
- 产品码
- 记录编号
- 面别/件别
- 型号
- 原图/结果图
- ROI 明细
- 异常原因

### 8.6 结果渲染优先级

- 现场是更重视“看最终结果”
- 还是更重视“看问题位置”
- 还是更重视“看统计摘要”

## 9. 第一版产品抽象建议

如果后续要从“标准产品 + 项目定制”角度继续推进，我建议第一版就按三层抽象：

### 9.1 标准底座层

统一保留：

1. 图像输入
2. 图像处理
3. 图像检测
4. 记录结构
5. 结果查看器
6. 模型/相机/存储管理

### 9.2 项目配置层

项目差异主要通过配置解决：

1. 图像数量与顺序
2. ROI 策略
3. 检测项与模型映射
4. 判定规则
5. 追溯字段
6. 渲染块选择

### 9.3 项目流程层

少数复杂项目通过流程编排解决：

1. PLC/IO 触发与回传
2. 中间动作联动
3. 翻面/旋转/移动
4. 复检回路
5. 整批逐张处理
