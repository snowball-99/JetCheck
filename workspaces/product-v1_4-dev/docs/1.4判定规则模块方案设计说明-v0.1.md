## 需求清单

### 需求 1：检测结果判定

#### 1. 需求说明

##### 1.1 背景

当前检测工具已经能够输出图像获取、图像处理、图像检测的结果，但业务上真正需要的是一个最终判定结论，例如 `OK / NG`。  
判定规则模块的作用，是在检测配置完成后，为这些结果补上一层统一的 `OK` 判定逻辑。

##### 1.2 目标

1. 让常见规则可以通过可视化方式完成配置。
2. 让复杂规则仍然可以通过表达式方式兜底。
3. 让规则配置和前面 3 步的结构保持一致，避免结构变化后误用旧规则。
4. 将用户配置聚焦在 `OK` 条件，降低页面复杂度。

##### 1.3 范围

1. 当前版本只覆盖 `OK` 判定条件，不单独开放 `NG / PENDING` 条件配置。
2. 当前页面分为两种模式：
   1. 可视化配置
   2. 表达式编辑
3. 两种模式互斥，一次只能使用一种。

##### 1.4 当前层级定义

当前判定对象按以下层级组织：

1. 总结果
2. 图像
3. 处理实例
4. 检测项

其中：

1. `总结果`
   代表整套工具最终的 `OK` 汇总结果。
2. `图像`
   代表某个图像获取实例的整体结果。
3. `处理实例`
   代表某个图像处理结果下多个检测项的聚合结果。
4. `检测项`
   代表某个单独检测模型实例的业务判定条件。

##### 1.5 当前开放配置范围

当前可视化模式下：

1. `总结果` 可查看，不可编辑。
2. `图像` 可查看，不可编辑。
3. `处理实例` 可编辑。
4. `检测项` 可编辑。

这样设计的原因是：

1. 常见业务规则主要落在“单个检测项怎么判”。
2. 复杂场景通常落在“一个处理实例下多个检测项怎么组合判”。
3. 总结果和图像层先提供摘要展示，避免上层规则编辑过重。

#### 2. 方案描述

##### 2.1 页面入口

1. 判定规则作为工具搭建的第 4 步。
2. 位于“图像获取 / 图像处理 / 图像检测”之后。
3. 用户在前面 3 步完成基础配置后，进入这一页补充业务判定逻辑。

##### 2.2 模式切换

1. 页面只保留一个模式切换入口。
2. 当前为可视化模式时显示“切换表达式编辑”，当前为表达式模式时显示“切换可视化配置”。
3. 点击后弹出确认提示。
4. 确认后清空当前模式配置，并恢复目标模式的默认规则。
5. 两种模式互斥，不做双向同步。

##### 2.3 恢复默认

1. 顶部提供“恢复默认”按钮。
2. 点击后基于当前工具结构重新生成一套默认判定配置。
3. 该操作既用于首次生成，也用于规则失效后的重建。

##### 2.4 默认生成规则

1. 默认规则会在以下时机生成或重建：
   1. 首次进入“判定规则”步骤时。
   2. 当前工具还没有判定规则配置时。
   3. 用户点击“恢复默认”时。
   4. 用户切换可视化配置与表达式编辑模式并确认后。
2. 默认规则会基于当前工具结构自动生成，不依赖手工逐条创建。
3. 默认生成原则如下：
   1. 总结果默认汇总所有处理实例结果。
   2. 每张图像默认汇总该图像下所有处理实例结果。
   3. 每个处理实例默认汇总其下所有检测项结果。
   4. 每个检测项默认生成一条 `OK` 判定规则。
   5. 不同检测类型使用不同的默认模板。
4. 总结果当前不开放编辑，但系统内部会自动生成默认汇总逻辑：
   1. 默认关系为“全部满足”。
   2. 默认参与对象为当前工具下所有处理实例。
   3. 该逻辑用于生成总结果摘要和默认表达式。
5. 图像当前不开放编辑，但系统会自动生成只读汇总逻辑：
   1. 默认按该图像下全部处理实例“全部满足”汇总。
   2. 该逻辑用于生成图像层摘要。
6. 处理实例默认规则如下：
   1. 常规场景默认关系为“全部满足”。
   2. 当一个处理实例下挂载多个检测项且全部为尺寸类型时，默认关系为“仅一满足”。
   3. 该默认规则适用于多尺寸模板互斥匹配场景。
7. 检测项默认规则如下：
   1. 尺寸类型：默认生成一条尺寸判定规则，默认关系为“全部满足”，尺寸项直接读取尺寸模板包输出，上下限优先读取预置示例值，没有则留空。
   2. 分类类型：默认生成一条分类判定规则，默认关系为“全部满足”，默认条件为模型第一个类别数量等于 1。
   3. 缺陷类型：默认生成一条缺陷判定规则，默认关系为“全部满足”，默认条件为“缺陷 数量 = 0”。

##### 2.5 可视化配置

1. 可视化配置采用左右布局。
2. 左侧为判定对象树，右侧为当前对象的内容区。
3. 左右区域分别内部滚动，不依赖整页滚动。

##### 2.6 判定对象树

1. 左侧展示总结果、图像、处理实例、检测项四类对象。
2. 所有对象都可点击，用于切换右侧展示内容。
3. 当前只有处理实例和检测项开放编辑。
4. 总结果和图像只提供只读查看。
5. 当前选中项高亮显示。
6. 可编辑项与只读项通过样式区分，不额外增加文字标签。

##### 2.7 本项OK条件

1. 所有对象右侧顶部都固定展示“本项OK条件”灰底摘要区。
2. 摘要用于把当前条件翻译成更接近业务语义的说明。
3. 总结果、图像、处理实例偏向汇总型表达。
4. 检测项偏向条件型表达。
5. 长文本自动换行，不使用横向滚动。

##### 2.8 总结果与图像

1. 总结果当前不开放编辑。
2. 点击总结果后，只展示总结果的 OK 条件如何由下层结果汇总得到。
3. 图像当前不开放编辑。
4. 点击图像后，只展示该图像的 OK 条件如何由下层结果汇总得到。
5. 这两层的主要作用是帮助用户理解整套规则结构，而不是直接在此层修改规则。

##### 2.9 处理实例

1. 处理实例层只负责定义“多个检测项之间是什么关系”。
2. 右侧提供关系选择和只读检测项列表。
3. 关系支持“全部满足 / 任一满足 / 仅一满足”。
4. 适合处理多个检测项全部通过、任一通过、或多模板互斥成功等场景。
5. 下方列表只展示参与判定的检测项，不支持在此层手动勾选或排除子项。

##### 2.10 检测项 - 分类类型

1. 分类检测项围绕“类别数量条件”进行配置。
2. 页面支持多条件。
3. 条件之间可配置为“全部满足”或“任一满足”。
4. 每条条件包含类别名称、条件类型、条件值三部分。
5. 用户可以新增和删除条件，用于表达多个类别的组合要求。

##### 2.11 检测项 - 缺陷类型

1. 缺陷检测项的交互结构与分类一致。
2. 类别固定为“缺陷”。
3. 用户主要配置缺陷数量条件及多条件关系。

##### 2.12 检测项 - 尺寸类型

1. 尺寸检测项围绕尺寸模板包输出的固定尺寸项进行配置。
2. 用户不能新增、删除或改名尺寸项。
3. 用户只能为每个尺寸项填写上下限。
4. 尺寸项之间支持“全部满足 / 任一满足”两种关系。
5. 适合表达全部尺寸合格或部分尺寸触发即可通过的场景。

##### 2.13 表达式编辑

1. 进入表达式模式后，只保留整套规则的表达式编辑区。
2. 不再展示判定对象树和单项配置区。
3. 页面只显示“语法检查通过 / 语法检查不通过”两种状态。
4. 每次输入都会触发基础语法校验。
5. 语法错误时同步体现到状态和输入框样式。

##### 2.14 语法校验

1. 当前只做基础结构校验。
2. 校验范围包括“路径:表达式”格式、括号配对、引号闭合、空参数、函数括号等。
3. 校验目标是拦截明显输入错误。
4. 当前不覆盖完整业务语义检查。

##### 2.15 规则失效提示

1. 当上游结构发生变化时，判定规则会被标记为失效。
2. 典型场景包括删除步骤、切换模型、调整处理关系、检测项结构变化等。
3. 页面会给出红色提示。
4. 用户需通过“恢复默认”重新生成规则。

#### 3. 适用场景

##### 3.1 当前适用的场景

1. 单个检测项的 `OK` 条件配置。
   1. 分类数量判断。
   2. 缺陷数量判断。
   3. 尺寸上下限判断。

2. 一个处理实例下多个检测项的组合判定。
   1. 全部检测项都需要通过。
   2. 任一检测项通过即可。
   3. 多模板互斥成功。

3. 基于当前工具结构自动生成一套默认规则，再在可视化界面上细化调整。

4. 可视化规则不足以表达时，通过表达式编辑进行整体兜底。

##### 3.2 典型适配示例

示例 1：分类检测

1. 场景：卡扣检测需要判断“卡扣已安装”数量为 1。
2. 适配方式：
   1. 在分类检测项下配置一条条件。
   2. `类别名称 = 卡扣已安装`
   3. `条件类型 = 数量=`
   4. `条件值 = 1`

示例 2：缺陷检测

1. 场景：焊点检测需要判断缺陷数为 0。
2. 适配方式：
   1. 在缺陷检测项下配置一条条件。
   2. `缺陷`
   3. `数量 = 0`

示例 3：尺寸检测

1. 场景：一个尺寸模板输出多条固定尺寸项，每条尺寸项都要配置上下限。
2. 适配方式：
   1. 在尺寸检测项下填写每个尺寸项的下限和上限。
   2. 选择“全部满足”或“任一满足”。

示例 4：多模板互斥

1. 场景：同一处理实例下挂两个尺寸模板检测项，正常情况下只允许一个模板匹配成功。
2. 适配方式：
   1. 在处理实例层将关系配置为“仅一满足”。

##### 3.3 当前不适用或不作为主支持范围的场景

1. 在可视化模式下同时维护多条规则。
2. 单独配置 `NG / PENDING` 条件。
3. 在总结果层和图像层直接做复杂可视化编辑。
4. 在处理实例层勾选、排除某个检测项。
5. 尺寸项自由增删或手工创建。
6. 表达式与可视化规则的双向同步。
7. 表达式的完整业务语义校验。
8. 复杂跨对象自由引用的高级规则编排。
