# JetCheck 1.4 尺寸检测方案决策清单 v0.1

本文档用于沉淀 1.4 尺寸检测方向中“需要先统一口径、再展开详细方案”的关键问题。

当前版本不是最终方案，而是用于帮助快速收敛方向的决策底稿。

## 0. 已确认口径（2026-04-08）

基于本轮讨论，以下内容先作为当前已确认口径：

1. 1.4 的尺寸检测能力应作为独立的本地工具存在，但需要和现有 JetCheck 产品融合。
2. 融合点至少包括：
   - 交互语言与整体风格保持同一产品体系感
   - 提供便捷启动方式
   - 尽量适配现有账号体系
   - 尺寸工程产出的尺寸方案 / 发布版本能够与客户端打通
3. 尺寸测量工具只负责编辑态，不承接产线实时运行。
4. 正式产线运行仍由现有 JetCheck 运行侧主导。
5. 尺寸测量工具需要考虑加入一定的验证能力，用于在尺寸方案配置完成后做发布前确认。
6. 正式检测产生的尺寸检测结果，需要进入 JetCheck 现有检测记录体系。
7. 1.4 一期范围中：
   - 测量值输出属于确定范围
   - 超差判定暂未最终拍板，继续评估
   - 业务综合结果不在本期范围内，因为 JetCheck 已有自己的业务结果定义体系
8. 交互入口暂未拍板，需要继续思考“本地安装后如何直接打开，以及平台端 / 客户端是否需要提供调起入口”。
9. 由于尺寸检测的图像来源是相机，因此像素尺寸标定不是可选项，而是 1.4 方案中的基础能力之一。
10. 标定能力需要同时影响：
   - 工程编辑与验证
   - 尺寸方案 / 发布版本发布
   - JetCheck 运行前校验
   - 检测记录追溯
11. 当前理想的标定方式不是开放式自由标定，而是产品预置若干种标准标定板类型。
12. 对这些标准标定板，系统内置对应的标定算子和流程，用户主要通过选择具体标定板类型来完成标定。
13. 一期先按“有限几种标准标定板 + 后续可扩展”设计，避免把标定能力做成高门槛专家工具。
14. 对现场变化的处理策略先按保守路线设计：
   - 能自动恢复的自动恢复
   - 需要重标定的引导重标定
   - 超出允许范围的及时阻断
15. 不把“标定”设计成万能兜底能力，需要明确区分：
   - 找正问题
   - 标定更新问题
   - 安装异常问题
16. 用户视角不需要理解复杂的“标定版本管理”，系统内部保留 `标定记录` 用于追溯即可。
17. 文档中原先的“位置补正”暂统一改称 `找正`。
18. 当前对找正的理解是：在搜索 ROI 内基于参考特征做匹配，并支持一定范围的平移 / 旋转恢复。
19. 文档中原先偏模糊的“模板”暂统一改称 `尺寸方案`，客户端实际消费的可运行产物叫 `发布版本`。
20. 一期标准标定板建议直接预置 2 到 3 种类型，可以是不同尺寸或不同样式，但整体保持可扩展。

## 1. 当前理解

基于当前输入，先形成以下前提：

1. 1.4 的重点不是继续扩展现有平台端训练框架，而是补充产品对尺寸检测场景的支持。
2. 尺寸检测能力倾向于采用“独立本地小工具”的形态，而不是完全塞进现有训练体系。
3. 该小工具不能成为完全割裂的新产品，仍需和 JetCheck 现有客户端能力融合。
4. 参考方向包括：
   - 基恩士 TM-X / TM-X Navigator 一类产品形态
   - 团队已有的“尺寸测量工具” demo
5. 1.3 当前已有少量尺寸检测相关表达，但还只是把“尺寸”作为一种结果类型，而不是完整的尺寸测量产品能力。

## 2. 当前可先接受的工作假设

在没有进一步拍板前，建议先采用以下工作假设推进：

1. JetCheck 继续承担账号、客户端、相机资源、运行集成、记录追溯等主产品职责。
2. 尺寸检测新增一套相对独立的“尺寸工程编辑能力”，用于搭建测量流程、找正、标注与公差。
3. 尺寸测量工具优先服务本地离线配置与本地运行，不以云端训练闭环为一期前提。
4. 1.4 一期的目标不是做成完整的视觉测量平台，而是先打通“能配置、能运行、能回传结果、能展示记录”的最小闭环。

## 3. 建议优先确认的决策项

以下问题建议优先拍板，因为它们会直接影响后面的信息架构、demo 方向和技术边界。

### 3.1 产品形态

#### 决策 1：1.4 的尺寸检测能力，以什么形态进入 JetCheck 体系？

当前状态：

- 已基本确认。

可选方向：

1. 完全独立的新本地应用，JetCheck 仅做结果接入
2. JetCheck 客户端内的新一级模块
3. 独立本地工具 + JetCheck 客户端内提供入口与运行集成

当前建议：

- 优先考虑第 3 种，即“独立本地工具 + JetCheck 融合入口”。

当前确认结果：

- 采用“独立本地工具 + JetCheck 融合”的方向。
- 不作为完全独立于 JetCheck 体系之外的新产品。
- 融合重点不仅是数据打通，也包括交互语言、视觉风格、启动方式和账号体系衔接。

原因：

1. 符合“本地小工具”的产品直觉，更适合承载尺寸测量这类工程化编辑体验。
2. 不会强行把尺寸测量编辑器塞进现有检测工具页面，避免交互结构失真。
3. 仍能保留 JetCheck 的统一账号、项目、运行、记录和现场使用链路。

需要你拍板的核心点：

- 后续再细化“融合深度”和“启动方式”，但产品形态方向本身已基本确认。

### 3.2 融合边界

#### 决策 2：JetCheck 和尺寸测量工具分别负责什么？

当前状态：

- 已基本确认。

当前建议边界：

JetCheck 负责：

1. 客户端登录、设备与项目上下文
2. 相机资源管理
3. 运行触发与外部系统对接
4. 统一记录、导出和追溯
5. 在运行页展示尺寸结果

尺寸测量工具负责：

1. 尺寸工程创建与编辑
2. 参考图像、标注与测量链路配置
3. 公差、测量项、判定规则配置
4. 输出可供运行时调用的尺寸方案发布版本或运行包

当前确认结果：

1. 尺寸测量工具负责编辑态，不负责正式产线实时运行。
2. JetCheck 继续作为正式运行载体。
3. 尺寸工具产出的尺寸方案 / 工程结果，需要能像当前“平台模型 -> 客户端模型”那样，与客户端形成可获取、可引用关系。

需要你拍板的核心点：

- 编辑态 / 运行态分工已确认，后续重点转为“尺寸方案如何发布、同步和被客户端消费”。

### 3.3 编辑态与运行态是否分离

#### 决策 3：尺寸检测是否要明确区分“工程编辑态”和“现场运行态”？

当前状态：

- 已确认。

当前建议：

- 明确分离。

原因：

1. 尺寸测量通常存在尺寸方案、标注、参考边、距离线、公差等强工程属性，不适合和一线运行页面混合。
2. 编辑态更接近专业配置工具，运行态更接近标准 JetCheck 的现场执行页面。
3. 这种分离也更贴近 TM-X Navigator 一类产品的使用心智。

需要你拍板的核心点：

- 默认按“一线操作员不进入编辑器，只消费已发布测量方案”继续设计；如后续出现调机、验收类场景，再补充例外角色。

### 3.4 工程对象模型

#### 决策 4：尺寸检测是不是要引入独立的“工程”概念？

当前状态：

- 已基本确认。

当前建议：

- 要。

建议对象层级：

1. 尺寸工程
2. 工程内的尺寸方案 / 标定配置 / 测量项
3. 工程发布产物
4. JetCheck 运行工具对发布版本进行引用

这意味着后续我们要明确：

1. 一个 JetCheck 检测工具是否可以关联一个或多个尺寸工程
2. 工程是按产品型号管理，还是按检测工位管理
3. 工程发布后是否允许保留多个发布版本

需要你拍板的核心点：

- 后续重点确认“工程发布产物”的形态，以及客户端侧看到的是“工程”“尺寸方案包”还是“可运行版本”。

### 3.5 图像输入由谁掌控

#### 决策 5：相机与取图链路归谁管理？

当前状态：

- 暂按已确认推进。

当前建议：

- 仍由 JetCheck 管理相机资源，尺寸测量工具主要消费图像样本和运行输入。

原因：

1. 现有 JetCheck 已经有相机和参数组能力，不值得在尺寸工具里再造一套。
2. 相机归 JetCheck 后，尺寸工具可以更聚焦在尺寸工程本身。
3. 后续分类、缺陷、尺寸三类能力也更容易在一套客户端资源体系下共存。

需要你拍板的核心点：

- 当前先按“尺寸工具不承接相机管理”推进；如后续验证环节需要取实时图，再补充最小必要能力。

### 3.5A 标定与像素尺寸换算

#### 决策 5A：像素尺寸标定由谁负责，作为什么对象存在？

当前状态：

- 新增关键问题，需重点展开。

为什么这是关键问题：

1. 尺寸检测最终需要把像素距离换算成真实物理尺寸。
2. 图像来源既然是相机，就不能只讨论“尺寸方案”而不讨论“标定”。
3. 如果标定归属不清，后续工程配置、方案发布、运行校验和记录追溯都会变得不稳定。

当前建议：

- 采用“JetCheck 管相机资源，尺寸工具管标定配置”的混合方式。
- 标定流程本身采用“标准标定板类型 + 内置标定算子 + 向导式流程”的产品化方案。
- 运行侧对现场变化采用保守策略，不默认用软件补偿兜住所有安装偏差。

建议边界：

JetCheck 负责：

1. 相机、镜头、参数组、分辨率等基础运行资源
2. 运行时读取当前实际取图配置
3. 在执行前校验当前配置是否满足发布版本所依赖的标定条件

尺寸测量工具负责：

1. 创建和维护标定配置
2. 把标定结果与尺寸工程 / 尺寸方案绑定
3. 在验证环节确认标定是否有效
4. 发布时输出标定相关依赖信息
5. 提供标准标定板选择、标定执行和标定结果确认流程

为什么不建议把标定简单当成“相机固定属性”：

1. 同一台相机在不同镜头、焦距、工作距离、安装姿态下，像素尺寸关系可能不同。
2. 同一台相机切换不同参数组、分辨率、裁剪范围后，也可能影响测量结果可用性。
3. 标定天然服务于“某个尺寸工程在某种成像条件下成立”，它和工程上下文关系更强。

建议新增的对象概念：

1. 标定配置
   - 描述像素到物理尺寸换算所依赖的标定资源与适用条件
2. 标定适用条件
   - 关联相机、参数组、分辨率、ROI 或工位条件
3. 标定记录
   - 系统内部保留，用于追溯某次具体标定结果
4. 标定板类型
   - 系统预置的标准标定板定义
5. 标定算子
   - 与某类标定板绑定的内置标定执行逻辑

当前建议的产品化方向：

1. 系统预置少量标准标定板类型，作为一期可选项。
2. 每种标定板类型绑定固定的标定算子和推荐操作流程。
3. 用户在界面上主要做的是：
   - 选择标定板类型
   - 采集或导入标定图像
   - 确认识别结果
   - 保存标定结果
4. 不要求用户手动配置复杂的标定算法参数，默认把复杂度收敛在内置算子里。
5. 后续如需扩展，再增加新的标准标定板类型和对应算子，而不是推翻整体交互模型。

这样设计的优点：

1. 学习成本低，更适合 JetCheck 当前目标用户。
2. 标定流程更容易标准化，减少现场误操作。
3. 发布、运行和追溯时也更容易校验“尺寸方案到底依赖哪一类标定能力”。

建议对现场变化建立三层判定模型：

1. 可自动恢复
   - 典型场景：工件位置变化、轻微平移、小范围旋转
   - 处理方式：优先通过找正恢复，不要求重标定
2. 需重新标定
   - 典型场景：工作距离轻微变化、成像比例变化，但仍在允许姿态范围内
   - 处理方式：通过标准标定板重新生成新的标定记录
3. 超出安装允许范围
   - 典型场景：相机角度变化明显、整体倾斜或偏移过大、透视关系被破坏
   - 处理方式：直接阻断运行，提示先调整安装，再重新标定

当前建议的产品原则：

1. 优先自动恢复，但只在可信范围内生效。
2. 需要重新标定时，只更新标定记录，不默认要求重新制作尺寸方案。
3. 超出安装允许范围时，宁可阻断，也不误导用户继续运行。

运行时建议增加的校验：

1. 当前相机是否匹配发布版本所绑定的相机 / 参数组约束
2. 当前取图分辨率、裁剪方式是否与标定条件一致
3. 当前现场变化属于“可自动恢复”“需重新标定”还是“超出允许范围”
4. 如果超出允许范围，默认阻断运行；如果属于可恢复范围，再允许继续执行

记录里建议保留的信息：

1. 使用的尺寸方案 / 发布版本
2. 使用的标定配置或标定记录
3. 当前运行时的相机 / 参数组信息

需要你拍板的核心点：

1. 你是否认可“标定配置”应成为独立对象，而不是只写进相机配置里。
2. 当前已倾向确认：运行时若发现变化超出允许范围，默认阻断而不是仅告警。
3. 一期是否至少做到“单一标定配置绑定单一尺寸方案”，暂不处理复杂多标定自动切换。
4. 一期预置的标准标定板类型准备先收敛成几种，以及这些类型是否需要和相机 / 镜头场景绑定推荐关系。

### 3.6 结果落在哪一套记录体系里

#### 决策 6：尺寸结果是独立记录，还是进入 JetCheck 现有检测记录体系？

当前状态：

- 已确认。

当前建议：

- 优先进入 JetCheck 现有检测记录体系，但扩展字段结构。

建议记录层级：

1. 总记录保留整次任务结果
2. 图像级记录保留每张图的结果
3. 子结果中新增尺寸测量项明细
4. 支持记录测量值、公差范围、是否超差、叠加图层信息

原因：

1. 现场使用更希望统一追溯，不希望分类/缺陷/尺寸各查一套记录。
2. 参考项目里也已经出现“同一次检测中既有分类结果也有尺寸结果”的组合诉求。
3. 这更符合 JetCheck 作为总运行载体的定位。

需要你拍板的核心点：

- 记录体系统一进入 JetCheck 已确认，后续重点转为“记录字段设计”和“运行页展示方式”。

### 3.7 尺寸判定范围

#### 决策 7：1.4 一期只做“测量值输出”，还是同时做“公差判定 + 业务结果”？

当前状态：

- 部分确认。

可选方向：

1. 只输出测量值
2. 输出测量值 + 超差判定
3. 输出测量值 + 超差判定 + 业务综合结果

当前建议：

- 一期至少做到第 2 种，尽量往第 3 种靠。

当前确认结果：

1. 测量值输出属于一期确定范围。
2. 超差判定暂不最终确认，继续评估。
3. 业务综合结果不在 1.4 一期范围内。
4. 原因不是尺寸场景不需要结果，而是 JetCheck 已经有一套独立的业务结果定义体系，本期不重复建设。

原因：

1. 只有测量值没有判定，现场闭环不完整。
2. JetCheck 现有运行与记录体系天然需要结果字段。
3. 多数项目最终还是需要“是否合格”的直接输出。

需要你拍板的核心点：

- 后续需要单独评估：如果一期不做超差判定，是否会影响方案验证、运行展示和记录可用性。

### 3.8 多测量项与复合场景

#### 决策 8：1.4 一期是否支持“一次任务内多个尺寸项并行测量”？

当前建议：

- 要支持，但先限定在“单图多测量项”或“同工件有限个测量项”的范围内。

原因：

1. 尺寸检测一旦真正落地，通常不会只有一个测量值。
2. 早期如果只按单测量项设计，后续数据结构和页面结构很容易返工。
3. 但一期也不建议一开始就把过于复杂的流程编排全部做满。

需要你拍板的核心点：

- 1.4 一期是否要直接考虑“双图 + 多测量项 + 混合判定”这类复合场景。

### 3.9 交互入口放在哪

#### 决策 9：用户从哪里进入尺寸工程编辑？

当前状态：

- 待继续打磨。

当前建议可选入口：

1. 客户端新增“尺寸工具”一级导航
2. 在“检测工具”内部新增“尺寸工程配置”入口
3. 在“模型管理”旁边新增“尺寸工程管理”

当前建议：

- 倾向于第 3 种，或者第 1 种。

原因：

1. 尺寸工程更像一种独立资源，不太像检测工具内部的一个普通步骤。
2. 如果直接塞进“检测工具”三步配置中，页面会很快失控。
3. 资源管理和工程发布更容易独立演进。

需要你拍板的核心点：

- 当前需要继续讨论的不是是否有入口，而是：
  1. 本地安装后是否允许用户直接独立打开
  2. 平台端是否提供调起入口
  3. 客户端是否提供调起入口
  4. 调起后是否要带账号、项目、尺寸方案上下文

### 3.10 一期交付目标

4. 配置必要的尺寸参数与可选公差信息

## 4. 当前推荐的总体路线

基于现有信息，当前推荐路线如下：

1. 产品定位：
   - 采用“独立本地尺寸测量工具 + JetCheck 融合”的路线。
2. 架构边界：
   - JetCheck 管运行与追溯，尺寸工具管工程编辑、标定、验证与发布。
3. 页面结构：
   - 客户端中新增尺寸工程相关入口，不强塞进原有检测工具三步搭建页。
4. 数据结构：
   - JetCheck 记录体系继续统一，但为尺寸结果补充测量值、公差、叠加层、标定记录等字段。
5. 一期目标：
   - 优先打通从工程编辑到运行记录的最小闭环。

## 5. 建议先不急着拍板的次级问题

1. 工程文件的具体目录结构
2. 测量项内部 DSL 或配置格式
3. 结果图层的具体绘制规范
4. 导入导出文件的精确格式
5. 发布包是否需要签名或校验
6. 是否需要兼容多种测量算法后端

## 8. 本轮讨论后仍待继续打磨的问题

基于当前输入，后续最值得继续展开的是以下 4 个问题：

1. 启动方式
   - 独立桌面入口、本地最近工程入口、平台端调起、客户端调起之间如何组合。
2. 账号与权限衔接
   - 是完全复用 JetCheck 账号登录态，还是允许工具本地免登录但在发布 / 同步时校验账号身份。
3. 方案打通机制
   - 客户端获取的是“工程列表”“尺寸方案”“发布版本”，还是“可运行测量包”。
4. 方案验证能力
   - 是只做本地单次试跑，还是支持小批量验证、误差统计、验证报告。
5. 标定模型
   - 标定配置是否独立管理，以及它和相机、参数组、尺寸方案之间的绑定关系。
6. 标定板体系
   - 一期内置哪些标准标定板，以及是否需要“新增标定板类型”的扩展机制。

## 7. 当前最建议你优先拍板的 3 个问题

如果希望先快速收敛，建议优先确认这 3 个问题：

1. 1.4 是“独立工具 + JetCheck 融合”，还是“JetCheck 内新模块”。
2. 尺寸工具是否只负责编辑态，运行态仍由 JetCheck 承接。
3. 尺寸结果是否统一进入 JetCheck 检测记录体系。
