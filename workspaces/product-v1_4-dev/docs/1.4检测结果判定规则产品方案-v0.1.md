# JetCheck 1.4 检测结果判定规则产品方案 v0.1

本文档用于承接“检测结果之后如何输出业务判定结果”这一层产品能力。

当前 JetCheck 已能输出检测结果，但还缺少一层稳定的业务判定机制：

1. 分类、缺陷、尺寸等原始结果如何转成 `OK / NG / PENDING`
2. 用户如何定义这些判定规则
3. 一期如何在保证能力的同时控制实现复杂度

这份文档先从产品设计角度给出方案，不直接展开规则引擎细节。

## 1. 目标与边界

### 1.1 目标

1. 在检测工具配置完成后，允许用户补充业务判定规则
2. 系统根据检测结果和规则输出最终判定结果
3. 一期覆盖分类、缺陷、尺寸三类检测的高频判定场景
4. 保留表达式编辑器作为高级兜底能力

### 1.2 不在一期解决的内容

1. 通用脚本系统
2. 任意跨节点自由引用
3. 深层嵌套规则编辑器
4. 复杂调试器和完整规则解释页
5. 面向最终客户开放 DSL 编写能力

## 2. 产品定位

判定规则不是新的检测流程，而是检测结果之后的一层配置能力。

整体链路应收敛为：

1. 配置图像获取、处理、检测工具
2. 生成检测结果结构
3. 配置判定规则
4. 输出判定结果

因此产品上建议将其理解为：

`检测配置 -> 判定配置 -> 运行输出`

而不是把判定规则拆成一套独立工具。

## 3. 一期产品原则

### 3.1 主交互不用表达式编辑器

表达式编辑器作为兜底方案保留，但不作为一期主交互。

原因：

1. DSL 对业务用户不友好
2. 表达式语法校验、回显、调试成本高
3. 一期直接暴露 DSL 容易把需求推成“脚本平台”

一期主交互建议使用：

1. 规则模板
2. 条件表单
3. 组合关系选择

底层再编译成规则 AST 或 DSL。

### 3.2 控制可配置层级

一期不建议把每一层节点都开放成自由规则配置。

建议用户主配置的层级只有两个：

1. 周期级判定
2. 检测项级判定

其余层级先按系统默认逻辑处理：

1. 图片获取默认以实例结果为准
2. 图片处理默认以实例结果为准
3. ROI 默认以内部检测结果聚合为准

这样可以显著降低规则树复杂度。

### 3.3 控制规则复杂度

一期建议限制如下：

1. 一条规则最多两层条件组
2. 条件关系只支持 `全部满足` 和 `满足任一项`
3. 不支持用户自定义函数
4. 不支持手工编写节点路径
5. 不支持跨图片、跨 ROI 的任意引用

## 4. 规则作用域设计

### 4.1 周期级规则

周期级规则用于输出最终 `CycleResult.result`。

典型场景：

1. 所有检测项都 OK，则周期 OK
2. 任一关键检测项为 NG，则周期 NG
3. 存在异常项则周期 PENDING

周期级规则面向的是“整次检测是否放行”。

### 4.2 检测项级规则

检测项级规则用于对某个分类检测、缺陷检测、尺寸检测的原始输出做业务判定。

典型场景：

1. 分类结果中“头像朝上”数量必须等于 1
2. 缺陷列表中“不允许存在毛刺”
3. 尺寸项“螺杆1”必须存在，且数值在区间内

检测项级规则面向的是“某项检测到底算 OK 还是 NG”。

### 4.3 为什么一期先不开放图片级和 ROI 级主配置

原因不是这些层不重要，而是：

1. 图片级和 ROI 级会显著增加配置入口数量
2. 一期主要业务判断仍集中在检测项和周期汇总
3. 图片级、ROI 级可以先由系统自动生成默认聚合逻辑

后续如果业务需要，再升级为：

1. 图片级规则
2. ROI 级规则
3. 高级路径规则

## 5. 页面结构建议

建议在检测工具配置完成后，新增一个独立的“判定规则”页面。

### 5.1 页面入口

推荐放置方式：

1. 作为检测工具配置完成后的独立页签
2. 位于“图像检测”之后
3. 页面名称直接叫 `判定规则`

不建议：

1. 塞到每个检测实例的小弹窗里
2. 藏在运行页里配置

### 5.2 页面整体布局

建议采用三栏布局：

1. 左栏：结果结构树
2. 中栏：当前节点规则列表
3. 右栏：规则编辑区

#### 左栏：结果结构树

作用：

1. 展示当前工具下的可判定对象
2. 让用户选择当前编辑对象

一期建议树结构显示：

1. 周期
2. 检测项列表

如果后续扩展，可继续加入：

1. 图片获取实例
2. 图片处理实例
3. ROI 节点

#### 中栏：规则列表

显示当前选中节点下已有规则：

1. 默认规则
2. 用户自定义规则
3. 规则状态
4. 命中后输出结果
5. 优先级

支持操作：

1. 新增规则
2. 编辑规则
3. 启用 / 停用
4. 删除规则
5. 调整优先级

#### 右栏：规则编辑区

当前选中一条规则后，在右侧编辑：

1. 规则名称
2. 规则类型
3. 条件组
4. 条件关系
5. 命中结果
6. 表达式预览

## 6. 一期规则编辑交互

### 6.1 主流程

推荐交互流程：

1. 用户进入判定规则页
2. 系统先自动生成默认规则
3. 用户选择“周期”或某个检测项
4. 根据检测类型选择规则模板
5. 填写参数
6. 保存后生成规则
7. 规则列表中可查看摘要

### 6.2 一条规则的基本组成

一条规则建议包含以下字段：

1. 规则名称
2. 作用域
3. 目标节点 ID
4. 条件组
5. 条件关系
6. 命中结果
7. 优先级
8. 是否启用
9. 规则来源
   - 系统默认
   - 用户自定义
   - 表达式高级模式

### 6.3 条件编辑控件

一期建议将原子条件控件控制在以下范围：

1. 结果状态比较
   - 结果等于
   - 结果不等于
2. 标签判断
   - 标签等于
   - 标签不等于
3. 数量判断
   - 数量等于
   - 数量大于等于
   - 数量小于等于
4. 数值判断
   - 数值大于
   - 数值小于
   - 数值在区间内
5. 异常码判断
   - errorCode 等于
   - errorCode 不等于

### 6.4 条件组合

一期只开放两种组合关系：

1. 全部满足
2. 满足任一项

交互上不要让用户看到 `all / any / countArray / allArray` 这些底层概念。

用户看到的是：

1. 条件之间关系：全部满足 / 满足任一项
2. 是否作用于“命中的结果列表”

## 7. 规则模板设计

规则模板是一期最重要的复杂度控制点。

建议系统按检测类型预置模板，用户通过选模板来配置规则。

### 7.1 分类检测模板

建议至少支持：

1. 指定标签数量等于 N
2. 指定标签数量大于等于 N
3. 指定标签数量小于等于 N
4. 只允许出现某个标签

示例：

1. `头像朝上` 数量等于 1
2. `有卡扣` 数量等于 1

### 7.2 缺陷检测模板

建议至少支持：

1. 总缺陷数等于 0
2. 指定缺陷数量等于 0
3. 指定缺陷数量小于等于 N
4. 不允许出现某类缺陷

示例：

1. `毛刺` 数量等于 0
2. `缺口` 数量小于等于 1

### 7.3 尺寸检测模板

尺寸检测建议单独处理，因为它不仅有数值，还有检测异常状态。

建议至少支持：

1. 指定尺寸项必须存在
2. 指定尺寸项 `errorCode = 0`
3. 指定尺寸项数值在区间内
4. 指定多个尺寸项全部合格

示例：

1. `螺杆1` 必须存在
2. `螺杆1` 异常码为正常
3. `螺杆1` 数值在 `9.0 ~ 10.0`

### 7.4 周期汇总模板

周期级建议至少支持：

1. 所有检测项结果都为 OK
2. 任一关键检测项为 NG，则周期 NG
3. 任一检测项为 PENDING，则周期 PENDING

## 8. 尺寸检测的特殊规则

尺寸检测不能简单等同于分类和缺陷。

一期建议固定一个产品原则：

1. 异常优先于数值比较

也就是：

1. 先判断尺寸项是否存在
2. 再判断 `errorCode`
3. 只有 `errorCode = 0` 时才进入数值比较

因此在尺寸模板中，建议系统自动串成下面的逻辑：

1. 目标尺寸项存在
2. errorCode = 0
3. value 在区间内

如果 `errorCode != 0`：

1. 默认判定为 NG 或 PENDING
2. 不再继续使用 value 做正常业务比较

## 9. 默认规则生成策略

为了降低首次配置成本，系统应在进入判定规则页时自动生成默认规则。

### 9.1 检测项默认规则

分类检测默认规则：

1. 若至少有一条结果，且命中模板条件，则输出 OK
2. 若未命中模板条件，则输出 NG

缺陷检测默认规则：

1. 若缺陷数量为 0，则 OK
2. 若缺陷数量大于 0，则 NG

尺寸检测默认规则：

1. 若存在异常码不为 0 的尺寸项，则 NG
2. 若关键尺寸项缺失，则 PENDING
3. 若尺寸值超出上下限，则 NG
4. 若全部关键尺寸项合格，则 OK

### 9.2 周期默认规则

建议默认生成：

1. 若所有检测项结果为 OK，则周期 OK
2. 若任一检测项为 NG，则周期 NG
3. 若无 NG 但存在 PENDING，则周期 PENDING

## 10. 表达式编辑器的定位

表达式编辑器作为兜底能力保留，但要严格控制其入口和作用范围。

### 10.1 入口位置

建议放在右侧规则编辑区底部：

1. 默认显示可视化规则配置
2. 提供 `查看表达式`
3. 提供 `切换高级编辑`

### 10.2 使用方式

高级编辑模式下：

1. 用户可以查看系统生成的表达式
2. 用户可以手动调整表达式
3. 系统提供语法校验
4. 系统提示“高级编辑后，可能无法完整回显到可视化表单”

### 10.3 产品边界

一期不建议：

1. 把高级编辑当主入口
2. 在普通页面大量暴露 DSL
3. 允许表达式跨越任意层级自由引用

## 11. 数据对象建议

虽然本文档重点是产品方案，但为了支持前端配置，一期的数据对象建议至少补以下字段。

### 11.1 结果节点

每个可判定节点建议至少有：

1. `id`
2. `label`
3. `type`
4. `path`
5. `result`
6. `children`
7. `availableFields`

其中：

1. `label` 用于展示
2. `id` 和 `path` 用于规则绑定
3. `availableFields` 用于驱动不同节点的条件编辑控件

### 11.2 规则对象

一期规则对象建议至少包含：

1. `id`
2. `name`
3. `scopeType`
4. `targetNodeId`
5. `ruleType`
6. `conditions`
7. `relation`
8. `onHitResult`
9. `priority`
10. `enabled`
11. `sourceType`
12. `expression`

## 12. 结果展示建议

规则解释页是否在一期做完整能力，可以后续再定。

但一期至少建议保留以下基础展示能力：

1. 当前检测项结果
2. 最终周期结果
3. 命中的规则名称

不一定要在一期做成完整解释页，但产品结构上要为后续解释能力预留字段。

## 13. 一期交付建议

如果按最小闭环推进，我建议一期交付内容收敛为：

1. 判定规则独立页面
2. 周期级规则配置
3. 检测项级规则配置
4. 分类 / 缺陷 / 尺寸三类规则模板
5. 默认规则自动生成
6. 高级表达式编辑器兜底入口
7. 最终判定结果输出到现有记录结构

## 14. 当前结论

1. 一期主交互应为模板化规则编辑，而不是表达式编辑
2. 表达式编辑器保留，但作为高级兜底能力
3. 规则配置层级先收敛到周期级和检测项级
4. 分类、缺陷、尺寸分别提供模板化配置
5. 尺寸检测必须体现“异常优先于数值比较”
6. 后续如有需要，再逐步扩展到图片级、ROI 级和更复杂的规则解释能力
